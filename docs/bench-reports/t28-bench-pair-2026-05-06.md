## Root Cause of Panic

Commit `7fc010c` (merged from master in `9661ff1`) added auto-rebuild of
bench-vs-mtcp with `--features fstack` when `/opt/f-stack/lib/libfstack.a`
is present on the bench host (AMI 1.0.15).

With `--features fstack`, `bench-vs-mtcp/build.rs` was calling
`pkg-config --static --libs libdpdk` to emit a second DPDK whole-archive
block — in addition to the DPDK already linked by `dpdk-net-sys` (transitive
dep via `dpdk-net-core`). This duplicated the `mbuf_dyn.o` object in the
final binary, causing `tailqinitfn_mbuf_dynfield_tailq` to run twice → panic.

T27 did not trigger this because the T27 bench-nightly.sh pre-dated the
auto-rebuild feature (`7fc010c` was not yet in the a10-perf branch).

## Fix

`a1ab7f3` — removed the redundant DPDK re-link from bench-vs-mtcp/build.rs
(fstack path). DPDK is satisfied by dpdk-net-sys; libfstack.a + system libs
are sufficient for F-Stack.
