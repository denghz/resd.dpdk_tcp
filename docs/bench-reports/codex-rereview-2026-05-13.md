# Codex re-review of publication-fix round (2026-05-13)

Verdict: STILL-BLOCKED

## Per-item assessment

### B1 - T57 to T58 regression
Yes, as a diagnosis/disclosure fix, not as a code fix. The new report gives a concrete environmental cause: all three stacks regressed together, so a dpdk_net-only code change cannot explain it (`docs/bench-reports/regression-diagnosis-2026-05-13.md:29-34`), and current ICMP/TCP probes put DUT-peer RTT in the 216-407 us band (`docs/bench-reports/regression-diagnosis-2026-05-13.md:40-56`). It also identifies ENA allowance counters and a stuck CPU hog as contributors (`docs/bench-reports/regression-diagnosis-2026-05-13.md:65-89`, `docs/bench-reports/regression-diagnosis-2026-05-13.md:91-134`). Residual: the recommended hard gates for active ENA throttling / CPU hogs are still recommendations, not implemented (`docs/bench-reports/regression-diagnosis-2026-05-13.md:229-243`).

### B2 - peer hardening
Mostly yes. Peer servers now set `TCP_USER_TIMEOUT=5000ms`, ignore SIGPIPE, and run one pthread per accepted connection in all three peer binaries (`tools/bench-e2e/peer/echo-server.c:29-38`, `tools/bench-e2e/peer/echo-server.c:94-156`; `tools/bench-e2e/peer/burst-echo-server.c:43-59`, `tools/bench-e2e/peer/burst-echo-server.c:261-280`; `tools/bench-vs-linux/peer/linux-tcp-sink.c:28-29`, `tools/bench-vs-linux/peer/linux-tcp-sink.c:111-133`). The suite preflight captures the two-ENI state and discloses linux_kernel uses a different ENI (`scripts/fast-iter-suite.sh:746-763`; `docs/bench-reports/methodology-and-claims-2026-05-09.md:49-57`). Fairness impact of 5s user timeout is consistent across all stacks because they all hit the same hardened peer; it is a recovery/liveness knob, not a normal-path timing path unless the DUT dies mid-stream. Caveat: the two-ENI confound is disclosed, not removed.

### B3 - RACK validation
Yes. The old vacuous `rack | tlp` low-loss pass was replaced with a dedicated `rack_reorder_4k` scenario that applies peer-ingress ifb reorder, temporarily enables peer SACK, uses a 4096B multi-segment payload, and asserts `tcp.tx_retrans_rack > 0` as ALL-of (`scripts/verify-rack-tlp.sh:24-39`, `scripts/verify-rack-tlp.sh:300-321`, `scripts/verify-rack-tlp.sh:428-459`, `scripts/verify-rack-tlp.sh:596-615`). Teardown restores qdisc and `tcp_sack` (`scripts/verify-rack-tlp.sh:462-478`, `scripts/verify-rack-tlp.sh:617-626`). Docs report three rack_reorder runs with 1965/1876/1802 RACK events (`docs/bench-reports/verify-rack-tlp.md:66-78`; `docs/bench-reports/t57-fast-iter-suite-fair-comparison-2026-05-12.md:268-276`). This directly addresses the original B3.

### I1 - fstack bimodality
No, not fully fixed. The investigation plausibly identifies F-Stack's 100us TX drain and bakes `pkt_tx_delay=0` into generated F-Stack config (`docs/bench-reports/fstack-bimodality-investigation-2026-05-13.md:13-31`; `scripts/fast-iter-setup.sh:231-238`). But T59 explicitly says bimodality persists after the fix, with fstack CV 15-22% at 128/256/1024B and payload_256 stuck in the ~300us mode across all 5 runs (`docs/bench-reports/t59-statistical-rigor-2026-05-13.md:169-180`, `docs/bench-reports/t59-statistical-rigor-2026-05-13.md:478-493`). The docs are honest, so this is not hidden, but the item cannot be considered fully closed for publication-quality RTT claims involving fstack.

### I2 - metric rename
Mostly yes. `Stack::throughput_metric_name()` now emits `pmd_handoff_rate_bps` for dpdk_net and `write_acceptance_rate_bps` for linux/fstack, and `throughput_is_wire_rate_calibrated()` returns false for all arms (`tools/bench-tx-burst/src/lib.rs:60-114`). Regression tests reject both wrong cross-arm names and the retired `throughput_per_burst_bps` label (`tools/bench-tx-burst/tests/burst_grid.rs:385-475`). I found no production code path assuming `throughput_is_wire_rate_calibrated()==true`; `rg` only finds the accessor, docs, and tests (`tools/bench-tx-burst/src/lib.rs:101-114`). Caveat: T59 says its data still contains legacy `throughput_per_burst_bps` because the rebuild did not pick up the rename, and the aggregator accepts that legacy name (`docs/bench-reports/t59-statistical-rigor-2026-05-13.md:231-235`; `scripts/aggregate-fast-iter.py:105-124`).

### I3 - statistical rigor
Partly. The N-run harness exists and records per-run seed dirs (`scripts/fast-iter-stats.sh:1-34`, `scripts/fast-iter-stats.sh:212-243`, `scripts/fast-iter-stats.sh:282-331`). The aggregator uses percentile bootstrap over per-run means, paired diffs, and paired Cohen's d (`scripts/aggregate-fast-iter.py:160-232`, `scripts/aggregate-fast-iter.py:543-583`). 1000 resamples is acceptable for a preliminary CI screen, and T59 clearly calls N=5 preliminary and recommends N=10+ (`docs/bench-reports/t59-statistical-rigor-2026-05-13.md:94-108`, `docs/bench-reports/t59-statistical-rigor-2026-05-13.md:372-400`). Caveat: pairing is by run index only (`scripts/aggregate-fast-iter.py:37-43`, `scripts/aggregate-fast-iter.py:553-566`); because stack order is shuffled within each run, the pair does not mean same ordinal position/time slot. Also, the T59 raw artifacts/AGGREGATE.md are not present under `target/bench-results/` in this worktree, so the report numbers cannot be independently re-derived from local data (`docs/bench-reports/t59-statistical-rigor-2026-05-13.md:112-117`; local listing showed no `stats-2026-05-13`).

### I4 - stack order randomization
Yes for eliminating fixed-order bias; not a perfect experimental design. The suite implements seeded Fisher-Yates per tool and records the matrix in `metadata.json` and SUMMARY (`scripts/fast-iter-suite.sh:398-451`, `scripts/fast-iter-suite.sh:490-519`, `scripts/fast-iter-suite.sh:1343-1375`). The refactor preserved per-stack arguments: dpdk/fstack still use the DPDK NIC/EAL/FSTACK_CONF, linux uses nsenter/peer ports, and reset happens after dpdk/fstack arms (`scripts/fast-iter-suite.sh:805-865`, `scripts/fast-iter-suite.sh:882-941`, `scripts/fast-iter-suite.sh:959-1023`, `scripts/fast-iter-suite.sh:1046-1105`). Bash `$RANDOM % 3`/`%2` has tiny modulo bias, but for 3 stacks and publication caveats it is good enough to break deterministic third-arm disadvantage (`scripts/fast-iter-suite.sh:412-440`). DPDK exclusivity is preserved by sequential arm dispatch and post-DPDK/fstack reset (`scripts/fast-iter-suite.sh:522-554`, `scripts/fast-iter-suite.sh:1552-1560`).
