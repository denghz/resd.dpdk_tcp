## Bug fixes this run (fstack)

Four successive bugs required iterative commits before all fstack results were
valid. Bugs 1–3 were fixed before the T50 run; bug 4 was discovered from T50
results and fixed post-run (`4163e7e`, 2026-05-09).

1. **`c60a501`** — `FF_EAGAIN`/`FF_EINPROGRESS` corrected from FreeBSD namespace
   (35/36) to Linux namespace (11/115). F-Stack writes Linux errno to
   `__errno_location`; confirmed by `errno=115` diagnostic in T50 dpdk burst log.

2. **`e699b57`** — `SOL_SOCKET`/`SO_ERROR` corrected from FreeBSD (0xffff/0x1007)
   to Linux (1/4). `ff_getsockopt` uses Linux-namespace constants; `ff_getsockopt_freebsd`
   exists separately. Also replaced direct `SO_ERROR` connect-detection with
   `ff_poll(POLLOUT, timeout=0)` gate — `SO_ERROR` alone returns 0 for both
   SYN\_SENT and ESTABLISHED, making them indistinguishable.

3. **`0eaa965`** — Added `WarmupRead` and `MeasureRead` echo-drain phases.
   Without draining the K-byte echo from the peer after each write burst, the TCP
   receive buffer fills → zero-window advertisement → deadlock. Validation:
   K=64KiB, 100 bursts: **581 Kbps → 13.8 Gbps** (23,800× improvement).

4. **`4163e7e`** (post-run, 2026-05-09) — Removed partial-read break in
   `pump_round_robin_once` echo-drain loop. The drain loop broke on
   `n < discard.len()` (partial `ff_read`), incorrectly treating a partial read as
   an empty recv buffer. On W×C combinations where the echo data arrived in multiple
   segments, the partial break left data stranded → recv buffer filled → zero-window
   → peer send buffer filled → zero-window back → `ff_write` EAGAIN for the full 60s
   window. Fix: break only on `n ≤ 0` (EAGAIN/EOF). Verification confirmed all three
   formerly-zero buckets produce non-zero goodput (2–5 Gbps range).

**Metric interpretation notes:**

`throughput_per_burst_bps` = K/(t1−t0) where t1 = all K bytes accepted into the
F-Stack TCP send buffer (`net.inet.tcp.sendbuf_max=16777216`, auto-resize). For
K≤1MiB, F-Stack accepts all K bytes in a single `ff_write` call (send buffer ≫
K), so t1−t0 ≈ a few µs of stack processing time, giving 9–13 Gbps — this is
the buffer-fill rate, not wire rate. dpdk\_net's equivalent metric is lower
because its send window forces multiple write+yield cycles through the DPDK poll
loop before all K bytes are transmitted.

`burst_steady_bps` = K/(t1−t\_first\_wire) for K≤1MiB is an artifact: when the
send buffer accepts all of K in one shot, t1−t\_first\_wire is near-zero and the
quotient diverges (raw values 26,000–140,000 Gbps). For K=4MiB and K=16MiB,
the send buffer fills and drains at wire speed, giving ~9–16 Gbps which reflects
actual F-Stack wire throughput.

`burst_initiation_ns` (t0, time from burst start to first byte on wire) grows
with K: 38 µs at K=64KiB vs 468–535 µs at K=16MiB. This reflects F-Stack's
internal send-buffer setup overhead, which grows with payload size.

## Notes

**dpdk\_net burst ETIMEDOUT at K≥1MiB large-G**: The burst bench crashes the
dpdk\_net binary at K=1024KiB,G=10ms burst 422 with
`tcp error during recv: errno=-110 (ETIMEDOUT)`. Diagnostic shows
`tx_data_mempool_avail=32588/32768` (pool not exhausted),
`conn_timeout_retrans=1` (single retransmit). The timeout is likely a transient
retransmit timer expiry under moderate NIC congestion rather than structural
pool exhaustion. K=4096KiB and K=16384KiB were never reached.

**fstack echo-drain architecture**: The three-bug fix sequence (errno namespace →
SOL\_SOCKET namespace → missing echo drain) is now documented in `fstack_ffi.rs`
and `fstack_burst.rs` module headers. The `WarmupRead`/`MeasureRead` state-machine
phases ensure the echo drain occurs before each subsequent burst write, preventing
TCP receive-buffer fill.

**mTCP stub**: mTCP driver remains a stub (exits 1). All 20 burst + 28 maxtp
buckets emit clean `bucket_invalid` rows with no crashes. No timeline for mTCP
real implementation given upstream dormancy (last meaningful commit 2021, DPDK
18.05/19.08 only).

**Linux maxtp vs dpdk\_net**: Even with the echo-backpressure caveat, the dpdk\_net
advantage is ~10–15× at C=64 large-W (2.1 Gbps vs 0.15 Gbps). A proper
comparison with a discard-sink peer would improve linux numbers at large-C but the
NIC-bypass advantage of dpdk\_net remains structural.

**fstack maxtp zero-goodput anomalies (resolved)**: Three buckets (W=16384B C=16,
W=65536B C=64, W=262144B C=16) produced 0.000 Gbps in the original T50 run.
Root cause (confirmed by commit `4163e7e`, 2026-05-09): the `pump_round_robin_once`
drain loop broke on `n < discard.len()` (partial `ff_read`), treating it as an empty
recv buffer. On affected W×C combinations, echo replies arrived in multiple segments;
the early break left data stranded → recv buffer fills → zero-window → peer stalls →
`ff_write` EAGAIN for the entire 60s window. Fix: break only on `n ≤ 0`. Verification
run (2026-05-09, 2s warmup / 10s window) confirmed fix: W=16384B C=16 → 2.49 Gbps,
W=65536B C=64 → 4.39 Gbps, W=262144B C=16 → 4.79 Gbps.

**fstack maxtp vs dpdk\_net throughput**: fstack peak (5.219 Gbps at W=16384B,C=4)
exceeds dpdk\_net peak (2.358 Gbps at W=4096B,C=4) by ~2.2×. Both use the same
ENA NIC at 2 Gbps line rate, so the discrepancy is measurement-methodology rather
than real wire throughput difference. fstack `sustained_goodput_bps` measures
bytes accepted by `ff_write` into the 16 MB send buffer (memory rate), while
dpdk\_net measures bytes transmitted on the wire. A fair comparison requires a
discard-sink peer that confirms delivered bytes. Numbers above the 2 Gbps NIC
link rate are buffer-fill artifacts.
