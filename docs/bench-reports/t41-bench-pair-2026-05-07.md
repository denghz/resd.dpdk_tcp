## Failure mode

C=16,W=16384 ran with 4/16 force-aborted after the 15s close timeout.
During close, `InvalidConnHandle` errors appeared for already-expired
handles, and `tx_drop_nomem=165,757,321` was logged — indicating the pool
was heavily exhausted during the 64-connection send storm. The accumulated
echo-server backlog from 16384-byte writes caused it to reject the
subsequent C=64,W=16384 connection attempts during handshake.

All remaining W=65536 and W=262144 buckets failed the same way
(open_persistent_connections failed on first connection), except
C=4,W=65536 which completed successfully — the 4-connection open succeeded
before the server backlog cleared fully.

## Pool state

Pool remained clean (32768) on all completed pre-open and post-close
snapshots; no mbuf leak. The tx_drop_nomem counter reflects pool pressure
during measurement (sends dropped when pool transiently empty), not leaks.

## Notes

Peer port confirmed open after run; echo-server recovered. The failure is
transient peer backlog exhaustion from a large burst of force-aborted
connections — not a DUT regression. Prior runs T33–T40 all completed
28/28 cleanly. A re-run (T42) should recover the full grid.
