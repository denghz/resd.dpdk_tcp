**Burst: dpdk arm clean.** All 20 dpdk_net buckets (K=64KB–16MB × G=0–100ms)
completed with valid data. fstack arm emits marker rows only (libfstack.a not
compiled in). 420 valid dpdk rows.

**Maxtp: dpdk arm failed starting at W=64B,C=16; linux arm fully clean.**
W=64B,C=1 and C=4 both reached the measurement phase and produced valid
goodput data, but `close_persistent_connections` timed out leaving 1 leaked
connection each. The two leaked connections consumed TX mbufs during the RTO
retransmit cascade; by W=64B,C=16 the TX data mempool dropped to 68/32768
(available/capacity), triggering `SendBufferFull` in the measurement phase.
Every subsequent dpdk bucket failed with `connect error: errno=-1` (echo-server
accept backlog flooded by ~16 leaked connections). Linux arm: 56 rows for all
32 buckets (W=64B–262KB × C=1–64) — fully clean.

**Root cause identified: missing window update ACK in close drain loop.**
The close drain loop calls `poll_once` (which moves bytes out of `recv.bytes`
via `deliver_readable`) but never sends a proactive ACK. When the bench sends W
bytes without reading echoes, the DUT receive buffer fills, rwnd drops to zero,
and the echo-server's blocking `write()` stalls indefinitely. With rwnd=0 and
no window update, the echo-server's FIN never arrives; `close_persistent_connections`
times out and the connection leaks its TX mbufs.

**Fix committed: `1ea2e3e`** — `engine.send_window_update(h)` added to the
Phase 2 drain loop (after `poll_once`, before `close_conn_with_flags`). This
sends an ACK with the recovered receive window on every drain iteration,
unblocking the echo-server's stalled `write()` so its FIN arrives and the close
completes.

## Root Cause Detail

`close_persistent_connections` Phase 2 drain loop:

```
poll_once()                        ← deliver_readable moves bytes out of recv.bytes
close_conn_with_flags(h, TW_SKIP)  ← sends FIN (if not already sent)
drain_and_accumulate_readable(...)  ← pops Readable events from event queue
```

`poll_once` internally calls `deliver_readable`, which moves received bytes from
`recv.bytes` into `delivered_segments` and opens the receive window.
**No ACK is sent** — `emit_ack` is only triggered by `TxAction::Ack` when an
incoming packet is processed; there are no incoming packets when the echo-server
is blocked in `write()`. Without a window update, the echo-server stays blocked,
its FIN never arrives, and the DUT stays in FIN_WAIT2 until the 15 s deadline.

## Fix: `1ea2e3e`

Added `pub fn send_window_update(&self, handle: ConnHandle)` to `Engine` in
`dpdk-net-core/src/engine.rs` (thin wrapper over private `emit_ack`). Called
from `close_persistent_connections` Phase 2 after `poll_once`:

```rust
engine.poll_once();
for &h in conns {
    // Proactive window update — unblocks echo-server write() stalled on rwnd=0.
    engine.send_window_update(h);
    let _ = engine.close_conn_with_flags(h, CLOSE_FLAG_FORCE_TW_SKIP);
    ...
}
```
