# T33 bench-pair report â€” 2026-05-07

## Summary

Root-cause identified and fixed: `snd_retrans` mbufs silently leaked on normal
TCP close, permanently depleting `tx_data_mempool` across the maxtp dpdk_net
grid.  After the fix all **28/28 WĂ—C buckets produce valid non-zero goodput**.

## Investigation trail

### Symptom
All maxtp dpdk_net grid buckets at or beyond C=64,W=256 produced 0 goodput.
`tx_data_avail` started at 32768, dropped to near 0 during C=64,W=256 warmup,
and never recovered â€” all subsequent buckets started with 0 pool mbufs.

### Diagnostic additions
Added `POOL pre-open / post-close` snapshots in `run_maxtp_grid_dpdk` and a
5-second post-close settling loop (later removed) to determine whether missing
mbufs trickle back with more NIC completion polling.

**Result**: 5 seconds of extra `poll_once` calls returned 0 additional mbufs â€”
they were permanently lost, not stuck in NIC completion queue.

### Root cause: `Mbuf` is Copy with no `Drop`

`Mbuf` (`mempool.rs`) is `#[derive(Clone, Copy)]` with no `Drop` impl.  The
normal-close path in `engine.rs` discards `TcpConn` via
`flow_table.borrow_mut().remove(handle)` â€” the returned `Option<TcpConn>` is
immediately dropped.  When `TcpConn` drops, its `snd_retrans.entries`
`VecDeque<RetransEntry>` is dropped, each `RetransEntry` is dropped, and `Mbuf`
is "dropped" â€” a no-op, silently forgetting the raw pointer without calling
`shim_rte_pktmbuf_free`.

The `force_close_etimedout` (abort_conn) path already did an explicit drain of
`snd_retrans.entries` before removing the conn.  The normal-close and
`reap_time_wait` paths did not.

### What was NOT the cause
- NIC TX completion stalling (the ENA `miss_txc_to=3` + `tx_done_cleanup` fix
  from the prior session handles that; it improved small-C recovery but did not
  fix large-C because the mbufs were genuinely dropped, not stuck in NIC CQ)
- Pool size too small (32768 is adequate; the leak inflated apparent in-flight)

### Fix (`engine.rs`)

In both the normal-close path (`handle_frame` outcome.closed branch, line ~4893)
and `reap_time_wait` (line ~3849): drain `conn.snd_retrans.entries` into a `Vec`
before `flow_table.remove`, then call `shim_rte_pktmbuf_free` on each â€” exactly
the same pattern `force_close_etimedout` already uses.

```rust
// Before flow_table.borrow_mut().remove(handle):
let retrans_to_free: Vec<RetransEntry> = {
    let mut ft = self.flow_table.borrow_mut();
    if let Some(conn) = ft.get_mut(handle) {
        conn.snd_retrans.entries.drain(..).collect()
    } else { vec![] }
};
for entry in retrans_to_free {
    unsafe { sys::shim_rte_pktmbuf_free(entry.mbuf.as_ptr()); }
}
```

### Notes on C=64 goodput dip at small W
C=64,W=256 shows ~0.32 Gbps vs ~0.83 for C=16,W=256. This is a genuine
throughput characteristic of the 256B write size with 64 connections under
pool pressure during warmup â€” not a measurement artifact. Pool exhaustion
during warmup (pre-measure avail=0) means send_bytes returns Ok(0) for most
calls during the measurement window, limiting effective goodput. The fix
ensures this does not leak into subsequent buckets.

### Close timeouts (normal â€” echo backlog)
Large-W, high-C buckets see 15s close timeouts with N connections force-aborted
(e.g., 27/64 for C=64,W=16384). This is expected: the echo server has O(GB) of
echoed data in its send buffer that our stack must receive before the peer sends
FIN. With the fix, abort_conn correctly frees those connections' snd_retrans
entries and pool recovers to 32768 in all cases.
