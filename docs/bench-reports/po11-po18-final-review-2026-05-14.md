## 2. Bugs Found

### LOW â€” PO11 log pattern very likely matches no logtype (no-op rather than wrong correctness)
File: `crates/dpdk-net-core/src/engine.rs:1571-1572`

The patch sets the level for pattern `"lib.eal.ena.com"`. The ENA driver registers its logtype via `RTE_LOG_REGISTER_SUFFIX(ena_logtype_com, com, WARNING)` in `drivers/net/ena/ena_ethdev.c:3931`, and DPDK's `drivers/meson.build:209` builds the name from `RTE_LOG_DEFAULT_LOGTYPE = pmd.<class>.<driver>` so the live logtype name is `pmd.net.ena.com`, not `lib.eal.ena.com`. Result: the call returns 0 but matches nothing, so the level does not change. Independently, the ENA logtype is registered at WARNING by default already, so the per-doorbell `ena_trc_dbg` is supposed to short-circuit inside `rte_log` regardless. Practical impact: the perf-win claimed for PO11 is illusory; observed wins come from PO12-PO18. The commit is not a correctness regression â€” the return value is intentionally ignored and a missed pattern is a benign no-op (confirmed by `rte_log_set_level_pattern` docs in `/usr/local/include/rte_log.h:147-150`: returns negative only on invalid `level`, not on unmatched pattern).

Repro sketch: build on any ENA host; run `RTE_LOG_LEVEL` introspection (or `dpdk-procinfo --log-dump`); confirm `pmd.net.ena.com` is still at WARNING (which it was already at) and `lib.eal.ena.com` does not exist as a registered logtype.

Mitigation: change the pattern to `"pmd.net.ena.com"` (or, defensively, both). Not a blocker for landing.

### LOW â€” PO18 stale docstring inside `send_bytes` still says `BULK_MAX = 32`
File: `crates/dpdk-net-core/src/engine.rs:6682-6687`

The block comment around the new `const BULK_MAX: usize = 96;` still narrates "Cap at BULK_MAX = 32" and "typical bursts are <=10 segments at MSS so the cap is rarely the limiter". The constant just below is 96. Reading the docstring next month will mislead. Pure documentation drift; not a correctness issue.

Mitigation: rephrase the 32-era comment in a follow-up.

No BLOCKER or HIGH-severity findings.

## 3. Per-PO Findings

### PO11 (`5257fdc`) â€” log-level pattern set after `rte_eal_init`
- Green: call is correctly sited AFTER `rte_eal_init` returns rc >= 0 (engine.rs:1540 â†’ 1571), wrapped in `unsafe`, uses `CString::new(...).unwrap()` on a static literal (no NUL panic risk).
- Green: return value ignored with explicit rationale; DPDK semantics confirm unmatched pattern is a no-op (`rte_log_set_level_pattern` only returns negative on bad `level`).
- Yellow (LOW finding above): pattern is `lib.eal.ena.com` but the real ENA logtype is `pmd.net.ena.com` per `drivers/net/ena/ena_ethdev.c:3931` + `drivers/meson.build:209`. The patch is benign-no-op on this host class.
- Operator-visible-warning concern: even if the pattern were correct, the new level is WARNING, so any genuine WARNING/ERROR/ALERT/CRIT/EMERG on the logtype is still visible. Cannot mask a real warning operators need.

### PO12 (`d04f9ae` + clippy follow-up `53951ce`) â€” RX-burst prefetch
- Green: prefetch loop bounds are safe. The pre-loop uses `mbufs[..n].iter().take(3)` (safe slice-then-take). The inline-loop uses `mbufs.get(i + 3)` on the full `[*mut rte_mbuf; 32]` array (engine.rs:3361) â€” returns `None` past index 31, and untouched slots in `[n..32]` are still `null_mut()` from init. `prefetch_mbuf_data` null-checks at `crates/dpdk-net-core/src/lib.rs:108-110` before any deref.
- Green: arch gates are present â€” x86_64 uses `_MM_HINT_T0`, aarch64 uses `_PREFETCH_READ + _PREFETCH_LOCALITY3` (mirror of T0). Other arches fall through to a no-op that discards the pointer to silence `unused_variables`.
- Green: `prefetch_mbuf_data` is now `unsafe fn` after `53951ce`; both call sites at engine.rs:3351 and engine.rs:3362 wrap the call in `unsafe {}` blocks with safety comments stating the rx_burst contract.
- Green: prefetch is a hint â€” even when given a non-NULL pointer to free / mid-cleanup memory the CPU is allowed to drop it (Intel SDM Vol 1 Â§11.7 / ARM ARM B2.10.3); no fault is generated by prefetch instructions on bad addresses.

### PO13 (`03be60e`) â€” Timer-wheel early-exit
- Green: `now_tick` derived from `now_ns / TICK_NS`. `crate::clock::now_ns()` is TSC-derived and monotone (clock.rs:42-50); TSC overflow is centuries away. `advance_last_tick` advances monotonically (only written inside `advance_timer_wheel` to the just-computed `now_tick`).
- Green: only two call sites of `advance_timer_wheel` (engine.rs:3288 idle path, engine.rs:3391 post-burst path); both inside `poll_once`. `pump_timers` test/abi path calls `fire_timers_at` directly without touching `advance_last_tick` (engine.rs:8817) â€” this asymmetry is fine because `fire_timers_at` has its own monotone gate inside the wheel (`tcp_timer_wheel.rs:151-152`).
- Green: missed-tick risk â€” `fire_timers_at` advances the wheel internally and unconditionally; only AFTER it returns does the engine set `advance_last_tick = now_tick`. If a fire handler panics, `advance_last_tick` stays at the previous value, so the next poll repeats the same tick. The wheel's own monotone gate then short-circuits the duplicate advance. No tick can be silently skipped.
- Yellow (informational, not a finding): the wheel's internal `last_tick` and the engine's mirror can briefly diverge under cascade (the wheel may advance internal `last_tick` past `now_tick` while the engine stores exactly `now_tick`). Next poll's tick comparison still catches up because TSC is monotone. SAFE.

### PO14 (`ca9f5d0`) â€” Bulk-alloc cache
- Green: refcount semantics. mbufs in the cache have `refcnt=1` (just bulk-alloc'd from `tx_hdr_mempool`). `tx_tcp_frame` pops the mbuf and either (a) hands it to `rte_eth_tx_burst` which consumes refcnt on success path; or (b) frees it directly on the `dst.is_null()` / `sent != 1` failure paths (engine.rs:2872, 2938). No leak window.
- Green: `Engine::drop` drains the cache via `rte_pktmbuf_free` at engine.rs:8243-8247 BEFORE the struct fields drop. Mempool is still alive because (i) the user `Drop::drop()` runs before field drop, and (ii) `tx_hdr_mempool` (declared at line 1188) is dropped via Rust's struct-field forward order, after `tx_hdr_cache` (declared at line 1244). Wait â€” actually `tx_hdr_mempool` is declared *before* `tx_hdr_cache`, so it drops first. That would be a hazard except `Drop::drop()` explicitly drains the cache before any field drop runs â€” so the drain returns mbufs to the still-live mempool. CONFIRMED SAFE.
- Green: reentrancy. The `borrow_mut()` on `tx_hdr_cache` lives only for the duration of `cache.pop()` inside a single block scope (engine.rs:2857-2863); the borrow drops before `shim_rte_pktmbuf_append`, `tx_offload_finalize`, and `rte_eth_tx_burst` run. So a re-entrant `tx_tcp_frame` (none observed in current code, but hypothetically from a TX-complete callback) would re-borrow_mut cleanly without panicking.
- Green: refill timing â€” only at end-of-poll under `#[cfg(not(feature = "test-server"))]` (engine.rs:3303, 3408). Sites cover both idle (no RX) and post-burst paths. If the cache empties mid-poll, `tx_tcp_frame` falls back to `shim_rte_pktmbuf_alloc` (engine.rs:2861) â€” same path as before PO14 â€” and bumps `eth.tx_drop_nomem` only if the alloc fails.
- Green: cache capacity 16, refill watermark 4. The constants are public-crate constants at engine.rs:1701-1707, used consistently in `refill_tx_hdr_cache_if_low` (engine.rs:3444). Stack array sized `TX_HDR_CACHE_CAP` so compile-time constant.
- Green: cached pointers are `NonNull<rte_mbuf>`. Bulk-alloc fills `[*mut rte_mbuf; CAP]`; only non-null slots are pushed via `NonNull::new(p)` (engine.rs:3464-3466). DPDK guarantees all-or-nothing for `rte_mempool_get_bulk` so on success every slot is non-null, but the `if let Some(nn)` is defensive belt-and-suspenders.
- Yellow (informational): test-server feature path gates refill out (engine.rs:3302-3303, 3407-3408) but still pops from the cache (engine.rs:2858). Result: under `test-server`, the cache always falls back to `shim_rte_pktmbuf_alloc`. The follow-up commit `7b70366` flushes PO15 counters on inject paths because of similar asymmetry. No correctness consequence.

### PO16 (`38e01b9`) â€” cached `tsc_hz` + per-poll `now_ns`

## 4. Cross-cutting

### Engine struct changes
- Added: `tx_hdr_cache: RefCell<Vec<NonNull<rte_mbuf>>>` (PO14, declared after `tx_pending_data`, line 1244).
- Added: `advance_last_tick: Cell<u64>` (PO13, line 1296), `tx_tcp_frame_bytes_acc: Cell<u64>` + `tx_tcp_frame_pkts_acc: Cell<u64>` (PO15, lines 1304/1307), `tsc_hz: u64` (PO16, line 1316), `poll_now_ns: Cell<u64>` (PO16, line 1326).
- No `unsafe impl Send` or `unsafe impl Sync` exists for `Engine` (verified by `grep -n "unsafe impl"` over `engine.rs` and `lib.rs` â€” zero hits). All new fields are `Cell`/`RefCell`-wrapped, which are `!Sync` by construction. The existing single-lcore `!Sync` invariant is preserved.

### Drop ordering
- `Engine::Drop::drop()` runs the user code first (engine.rs:8204 onward): clears `flow_table`, drains `tx_pending_data` ring (engine.rs:8226-8235), drains the new PO14 `tx_hdr_cache` (engine.rs:8243-8247), flushes PO15 counters (engine.rs:8251), takes the FaultInjector, then stops/closes the port.
- After `drop()` returns, fields drop in declaration order (top-down). `_rx_mempool` (line 1177) drops first; `tx_hdr_mempool` (line 1188) and `tx_data_mempool` (line 1189) follow. By that time, the PO14 cache `Vec` is empty (drained inside `drop()`) so returning the empty `Vec` to its allocator does not touch the mempool.
- Vec destructor does NOT free `NonNull<rte_mbuf>` elements (NonNull is `Copy`, no `Drop` impl), so even if the user `drop()` panicked partway through, the worst case is leaked mbufs â€” not a UAF. The explicit drain ensures mbufs are returned to the still-live mempool on the happy path.

### C ABI surface
- No new C ABI entry points added by PO11-PO18. The Box<Engine> opaque-pointer pattern at `crates/dpdk-net/src/lib.rs:51-63` is unchanged. C callers reach the engine via `&Engine`; new methods (`refill_tx_hdr_cache_if_low`, `flush_tx_tcp_frame_counters`) are `pub(crate)` so they are not exposed across the FFI boundary.
- `dpdk_net_flush` (`crates/dpdk-net/src/lib.rs:585-587`) now indirectly flushes the PO15 counter accumulators via `flush_tx_pending_data` â†’ `flush_tx_tcp_frame_counters`, preserving the previously-implicit invariant that a flush-from-the-outside leaves counters at the wire-truth value.

## 5. Test coverage gaps

- The library unit-test suite (`cargo test --release --package dpdk-net-core --lib --features test-server,test-inject -- --test-threads=1`) passes all 455 tests â€” confirmed in this review session.
- PO14 cache-drain path is not directly tested. A test that constructs an `Engine`, calls `tx_tcp_frame` 8+ times (to populate the cache), then drops the engine and verifies the mempool returns to full would catch a future regression where the drain step is removed. Today's coverage relies on Valgrind / DPDK's internal pool tracking â€” not part of the in-repo test loop.
- PO15 counter desync was caught by `counter-coverage::cover_eth_tx_pkts` / `cover_eth_tx_bytes` (per the `7b70366` commit message), which forced the inject-path flush. Future test-only RX paths that simulate one poll iteration must call `flush_tx_tcp_frame_counters` â€” there is no compile-time enforcement of this; an `#[must_use]` accumulator or a `debug_assert!` panic on engine drop with non-zero accumulator could enforce it. Not a blocker.
- PO16's `poll_now_ns == 0` defensive fallback is not exercised; the cache is set at every poll_once entry. Future maintainers may remove the fallback without a test failure. Adding a Rust unit test that constructs an Engine and calls `advance_timer_wheel()` without entering `poll_once` first would lock the contract.
- PO18 `BULK_MAX=96` with `total_segments > 96` falls into the partial-fill branch. The `assert!(bulk_n <= BULK_MAX)`-equivalent invariant is implicit. Today no test exercises bytes.len() large enough to clamp `total_segments` at 96. A focused test with `send_bytes(150 Ã— MSS)` would lock the partial-fill behavior.
