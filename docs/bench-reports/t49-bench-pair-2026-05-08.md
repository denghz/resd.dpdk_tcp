## Notes

**Peer degradation at large W**: W=65536 C=4/16 had "missing" connection
states at measurement time (2/4 and 6/16 respectively); at C=64 the peer
could not accept new connections. This is the same mechanism as the T26
`SendBufferFull` cascade but milder; the peer's echo recv buffer fills
under high-C large-W load before the DUT's close sequence completes.
Previous dpdk-only runs (T33–T48) did not exhibit this because each run
only stressed the dpdk_net path; the added linux 28-bucket pass increases
overall run wall-clock time, allowing the peer more time to accumulate
backlog connections.

**dpdk_net vs linux**: dpdk_net peaks at ~2.5 Gbps vs linux peak of
~0.19 Gbps — ~13× advantage from userspace NIC bypass.

**fstack stub**: all invalid rows are clean (no crashes, ff_init succeeds,
DPDK EAL + FreeBSD TCP stack initialises correctly). Real comparison data
pending peer AMI bake.

**First comparison run**: T49 validates the three-stack bench-vs-mtcp
pipeline end-to-end; dpdk_net + linux produce real data, fstack produces
clean invalid markers.
