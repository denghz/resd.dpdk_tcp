## Performance Gap Analysis

### Burst Initiation — dpdk_net is the clear winner at large K

At small bursts (K=64KiB) both stacks initiate at ~36µs: both are completing TCP handshake
and sending the first segment with equivalent latency. The engineering gap starts at K=256KiB
where fstack's BSD socket-buffer allocation becomes visible, and widens to 63× at K=4MiB.
This is a fundamental difference in architecture: dpdk_net writes directly from the application
buffer without a separate kernel copy or mbuf-chain construction step.

### Maxtp — dpdk_net vs fstack at wire rate

Where dpdk_net data is valid (C=1), the comparison at equal W:

| W | dpdk_net (Gbps) | fstack C=1 (Gbps) | note |
|---|---|---|---|
| 64B | 0.158 | 0.010 | 15.8× dpdk advantage |
| 1024B | 1.543 | 0.301 | 5.1× dpdk advantage |
| 16384B | 2.469 | 4.776 | fstack higher — **buffer-fill artifact** |
| 65536B | 2.264 | 5.121 | fstack higher — **buffer-fill artifact** |

For W≤1024B (below FreeBSD's mbuf threshold where buffer-fill doesn't inflate),
dpdk_net has 5–16× higher wire throughput at C=1.

At W≥4096B fstack's reported values exceed ENA link capacity (2.5 Gbps) and are
buffer-fill rates; wire delivery rate for fstack at large W is unmeasured in this run.

### Linux Drain Fix Analysis

**What the fix did:** Removing the partial-read early-exit allows the drain loop to exhaust
the socket RX buffer before writing each round. This improved C=16 and C=64 results from
near-zero to 0.027–0.275 Gbps.

**What the fix did not do:** The 15 remaining near-zero buckets (mostly C≤4, W≥256B)
are not a code bug — they reflect a measurement methodology limitation. The echo peer sends
back every byte we write. At low C, the round-robin loop revisits each connection faster than
RTT, so the drain gets WouldBlock before the echo arrives. As writes accumulate, the peer's
echo lags, its send buffer fills, the peer sends zero-window to us, and our TX send buffer
fills → writes return EAGAIN → goodput drops to near-zero.

dpdk_net is immune to this because its engine drops RX overflow (recv_buf_drops) and never
sends zero-window — the peer can always send echoes, so the peer always ACKs our data.

**Fair linux comparison requires a discard-only peer:** To measure linux TCP's raw TX
throughput without echo contamination, linux_maxtp should connect to a port that discards
received bytes rather than echoing them. A discard peer would allow linux kernel TCP to reach
near-line-rate throughput, providing a fair dpdk_net vs linux comparison.
