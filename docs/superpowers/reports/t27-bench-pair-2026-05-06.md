# T27 — bench-pair (2026-05-06) — persist timer validated

**Run:** `target/bench-results/2026-05-06T08-29-16Z/`
**Instance:** AMD EPYC 7R13, ENA, kernel 6.8.0-1053-aws, DPDK 23.11.0

**Started:** 2026-05-06T08:29:16Z
**Cost:** ~$1, ~75 min wall

## Goal

Validate RFC 9293 §3.8.6.1 persist timer (commit `fa703b5` / a10-perf `5efb72d`).
T25 target buckets — W={1024,4096,65536},C=1 — showed 0 Mbps due to zero-window
deadlock (tx_pps ≈ 1/60). All three should show non-zero throughput.

## Result: PASS — persist timer eliminates ZW deadlock at all three target buckets

| Bucket | T25 | T27 | Δ |
|--------|----:|----:|---|
| W=1024B, C=1 | **0 Mbps (ZW)** | **2,249 Mbps** | ✓ Fixed |
| W=4096B, C=1 | **0 Mbps (ZW)** | **3,608 Mbps** | ✓ Fixed |
| W=65536B, C=1 | **0 Mbps (ZW)** | **3,780 Mbps** | ✓ Fixed |
| W=16384B, C=1 | 708 Mbps | **4,188 Mbps** | +492% (persist + T21 combined) |

The persist timer also lifted W=16384,C=1 from 708 Mbps (T25, partial benefit of T21
fix but ZW on first recv-buf fill) to 4,188 Mbps (T27, zero-window probe prevents
permanent stall after the first fill cycle).

## dpdk_net vs linux at C=1 (persist-timer window)

| W_bytes | dpdk_net | linux | ratio |
|--------:|---:|---:|---:|
| 64 | 293 | 311 | 94% |
| 256 | 900 | 845 | **107%** |
| 1,024 | 2,249 | 2,156 | **104%** |
| 4,096 | 3,608 | 6,777 | 53% |
| 16,384 | 4,188 | 9,525 | 44% |
| 65,536 | 3,780 | 9,530 | 40% |
| 262,144 | 3,848 | 9,529 | 40% |

dpdk_net matches or beats Linux for small writes (W≤1024). The large-write gap (W≥4096)
is the expected TSO/LRO-absent overhead — dpdk_net sends one packet per 4096-byte write
while Linux coalesces. This gap was accepted in T24 and is unchanged.
