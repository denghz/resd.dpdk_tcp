## Root cause analysis — regressions vs T24

### Zero-window deadlock (ZW) at C=1 for W={1024, 4096, 65536}

All three show `tx_pps = 1/60` (exactly one packet per 60-second measurement window). This is the persist-timer fingerprint: the echo server's recv buffer hit zero-window at some point during warmup, `snd_wnd` dropped to 0, and since `snd_retrans` was empty when the zero-window was advertised, no RTO fire occurred. The DUT cannot send new data (`snd_wnd=0`) and has nothing to retransmit (`snd_retrans` empty), so it waits for a window update that never arrives.

**dpdk_net does not implement the RFC 9293 §3.8.6.1 persist timer** (the mechanism for sending periodic zero-window probes when `snd_wnd=0` and `snd_retrans` is empty). Without a persist timer, a zero-window stall deadlocks permanently.

**Was this caused by the T21 fix?** No. The code path for the DUT's advertised window (`free_space_total >> ws_shift_out`) is unchanged. The echo server's window advertisement to DUT (`snd_wnd`) is also independent of `rcv_wnd`. The T21 fix widened only the seq-window check (`in_window(rcv_nxt, seg.seq, rcv_wnd)`), which doesn't affect the send path.

The zero-window was triggered by the echo server's recv buffer filling during a burst, which is environmental — T24 avoided it by chance (different instance scheduling), T25 hit it. This gap was present in both T24 and T25 but only manifested in T25.

### SendBufferFull at high C (W=64/256/1024/4096/16384/65536/262144 at C=16 or C=64)

The TX data mempool (266240 mbufs) exhausts when many connections accumulate unACKed segments. This is a **pre-existing issue identical in T24** (T24 log shows same failures at W=64,C=16 and W=64,C=64). The connections are never closed between buckets (ConnHandle does not implement Drop-close), so old connections hold mbufs hostage until their retransmits complete.

### Connection accumulation and force_close

Without explicit per-bucket close, the engine accumulates all connections opened across all prior buckets. By the W=16384,C=4 bucket (~170 connections in the engine), accumulated connections from earlier failed buckets have stale in-flight data. When those connections' RTOs fire 3+ times, `force_close_etimedout` is called — which emits `InvalidConnHandle` errors to subsequent `send_bytes` calls if the handle was reused. This is also a **pre-existing issue visible in T24** (e.g., W=1024,C=64 failed with `InvalidConnHandle(44)` in T24).

## Recommendation

1. **Implement persist timer** (RFC 9293 §3.8.6.1) in the A11 roadmap. The zero-window deadlock is observable without artificial load — any single-connection high-throughput scenario that briefly hits zero-window will deadlock permanently.
2. **Add per-bucket connection close** to the bench harness so accumulated connections don't pollute subsequent buckets.
3. **Re-run bench-pair** after the persist timer is in place to get clean comparative data.

T21 validation objective is met: W=16384,C=1 improved from 0 to 708 Mbps.
