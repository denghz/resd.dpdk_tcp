## T21 root cause attribution (the smoking gun)

Two stall events captured by the new diag:

### Burst arm — K=262144 G=10ms, burst 3295
```
snd_una=1133254479 snd_nxt=1133254479 snd_wnd=0 send_buf_pending=0 send_buf_free=262144
srtt_us=533 rto_us=5000
| input_drops: bad_seq=1852 bad_option=0 paws_rejected=0 bad_ack=0 urgent_dropped=0
```

Stalled at burst 3295 (NOT burst 1) → emergent deadlock after 3294 successful bursts. Rejection rate ~10/s during the 180s deadline. Modest; consistent with peer-side flow-control deadlock pattern (DUT echo-server scenario at large K).

### Maxtp arm — C=1 W=16384, wedged for full 60s window
```
snd_una=1188157395 snd_nxt=1188157395 in_flight=0
snd_wnd=0 room_in_peer_wnd=0 srtt_us=111 rto_us=5000
| input_drops: bad_seq=5,555,029 bad_option=0 paws_rejected=0 bad_ack=0 urgent_dropped=0
```

**5.5 million bad_seq drops in 60 seconds = ~92,500 rejections/sec.** This is a tight-loop rejection storm. The engine receives peer segments and rejects 100% of them as out-of-window. Combined with `in_flight=0` (engine has zero unacked bytes — everything DUT sent was ACKed) and `snd_wnd=0` (peer advertised zero window), the connection is wedged with peer probably retrying / sending window probes / dup-ACKs that the engine's seq-window check rejects.

### Verdict against codex prediction table

| Hypothesis | Observed | Match |
|---|---|---|
| H1: snd_wnd-stuck (peer ACKs not arriving) | bad_seq=5.5M (peer IS sending, engine rejects) | ✗ |
| **H1': snd_wnd-stuck (engine drops peer's recovery segs)** | **`bad_seq` non-zero, snd_wnd=0** | **✓** |
| H2: snd_una-stuck | `bad_ack=0` | ✗ |
| H4: state-leakage | snd_una/snd_nxt look post-hand­shake-aligned | unlikely |

**Confirmed: T21 is a real engine input-validation bug.** When peer's recovery-side segments arrive (window probes, dup-ACKs, retransmit-of-our-already-ACKed-data), engine rejects them all as bad_seq. With C=1 there's no other connection to absorb traffic so the bucket wedges entirely.

The fix is in `tcp_input.rs::handle_established` around the bad_seq drop site (~line 779-787). Likely:
- The seq-window check is too strict for window probes (peer's seg.seq = peer_snd_nxt - 1, 1 byte payload to wake DUT — engine sees seq < rcv_nxt and drops)
- OR engine's rcv_nxt isn't being advanced correctly so subsequent peer segments fall outside the window

Diagnosis next: capture tcpdump on a failing C=1 W=16384 run, decode peer's segment headers vs engine's rcv_nxt at drop time. Or: add bad_seq sub-categorization (probe vs dup vs retrans) to the existing counter.
