## Goal

Validate the persist timer (RFC 9293 §3.8.6.1, commit `fa703b5`): W={1024,4096,65536} B,C=1
should no longer show 0 Mbps (zero-window deadlock) as observed in T25.

## Outcome: goal NOT met — close regression invalidated dpdk_net data

Commit `8cbd086` (landed 2 minutes before launch) added `close_persistent_connections`
between every maxtp bucket. This caused a cascading SendBufferFull failure through 26 of
28 dpdk_net buckets. Only the first two buckets (W=64B,C=1 and W=64B,C=4) produced data;
every subsequent bucket failed.

Root cause: on C=16/C=64 failures the TX data mempool was already exhausted. Calling
`close_conn` + 500 ms drain left FIN_WAIT_1 connections with stranded mbufs. The next
bucket (W=256B,C=1, which succeeded at 746 Mbps in T25) started with the pool still
depleted and immediately hit SendBufferFull.

Fix: `dc2fc42` — revert the two `close_persistent_connections` calls. ConnHandle has no
Drop impl, so dropping the Vec leaves orphaned connections in the engine; their
snd_retrans drains naturally during the next bucket's poll_once calls, matching T24/T25
behaviour.

## Close regression analysis

In T24 and T25, `ConnHandle` is intentionally not RAII — dropping the Vec leaves
connections live in the engine. On failure (e.g. SendBufferFull), orphaned connections
sit ESTABLISHED+idle, and their snd_retrans ACKs drain during the *next* bucket's
`open_persistent_connections` poll loop before new sends begin.

`close_persistent_connections` (8cbd086) broke this by:
1. Calling `close_conn` → engine sends FIN, enters FIN_WAIT_1
2. 500 ms drain — insufficient when TX data mempool is exhausted
3. FIN_WAIT_1 mbufs remain stranded beyond the drain window
4. Next bucket starts with a depleted pool → immediate SendBufferFull

The fix (dc2fc42) restores the original drop-silently semantics.
