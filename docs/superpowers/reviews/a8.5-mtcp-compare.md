## Findings

### Must-fix (correctness divergence)

*(none)*

A8.5 is a test-coverage expansion phase — no new wire-protocol algorithms were introduced, so there is no surface where we can be algorithmically wrong relative to mTCP. The shutdown subset, the test-FFI pumped wrapper, and the shim patches do not implement protocol behavior; they expose existing engine paths through narrower APIs. The TX option-order question (see AD-1) is corpus-compatibility, not correctness.

### Missed edge cases (mTCP handles, we don't)

*(none)*

Every input-side defensive check we examined in mTCP is either weaker than ours or absent:
- mTCP `ParseTCPOptions` has an unfixed `optlen - 2` underflow on unknown option kinds — our parser explicitly rejects (`OptionParseError::ShortUnknown`), with a comment naming the mTCP bug we defend against.
- mTCP `mtcp_write` doesn't NULL-check `buf` before calling `CopyFromUser` — our `dpdk_net_send` returns `-EINVAL` on `len > 0 && buf.is_null()`.
- mTCP `ProcessRST` carries an in-source TODO ("we need reset validation logic") — our RST window validation is in place and the new `tools/tcpreq-runner/src/probes/rst_ack.rs` exercises it.
- mTCP `icmp.c` only logs `ICMP_DEST_UNREACH` / `ICMP_TIME_EXCEEDED`; it does not act on `ICMP_FRAG_NEEDED` / update PMTU — we do (per RFC 1191) and the new ligurio `runnable-no-crash` bucket soak-tests the malformed-ICMP path.

There is nothing mTCP catches that we drop.

### Accepted divergence

- **AD-1** — TX option emission order: WSCALE-second vs WSCALE-last.
  - mTCP `mtcp/src/tcp_out.c:80-115` (`GenerateTCPOptions`): `MSS, [NOPs+]SACK_PERMIT, [NOPs+]TIMESTAMP, NOP+WSCALE` — WSCALE last, matching modern Linux 5.x kernel `net/ipv4/tcp_output.c::tcp_options_write`.
  - Ours `crates/dpdk-net-core/src/tcp_options.rs`: `MSS, NOP+WSCALE, SACKP, TS, NOPs+SACK-blocks` — WSCALE second.
  - **Resolution (option c):** kept current WSCALE-second order with explicit corpus-compatibility rationale. Added master spec §6.4 row `AD-A8.5-tx-wscale-position` (line 469). Updated `tcp_options.rs` module doc to correctly identify the order as shivansh/ligurio-corpus-matching rather than "Linux canonical".

- **AD-2** — `dpdk_net_shutdown` rejects `SHUT_RD` / `SHUT_WR` with `EOPNOTSUPP`.
  - mTCP `mtcp/src/include/mtcp_api.h`: there is **no `mtcp_shutdown()` public API** at all — only `mtcp_close()`. Half-close is not exposed.
  - Ours `crates/dpdk-net/src/lib.rs:1028-1059`: `SHUT_RDWR` dispatches to full close; `SHUT_RD` / `SHUT_WR` return `-EOPNOTSUPP`; invalid `how` returns `-EINVAL`.
  - **Resolution:** `AD-A8.5-shutdown-no-half-close` row (spec §6.4 line 467) updated to cite mTCP precedent ("`mtcp/src/include/mtcp_api.h` exposes no `mtcp_shutdown()` API at all — a stronger version of the same decision").

- **AD-3** — Pumped test-FFI wrapper (`dpdk_net_test_shutdown`).
  - mTCP: no equivalent. mTCP tests use sample applications calling the production `mtcp_*` API; corpus-style synchronous wire tests don't exist in their tree.
  - Ours `crates/dpdk-net/src/test_ffi.rs:291-307`: thin wrapper that calls `dpdk_net_shutdown` then drives one engine pump so the FIN lands in the TX intercept ring before packetdrill's next expectation matches.
  - **Resolution:** documented in T8 commit message + `dpdk_net_test_shutdown` doc-comment. The pumped-wrapper convention is a property of the synchronous-shim model (not the production API). No separate spec row needed.

- **AD-4** — `RunnableNoCrash` classifier verdict (crash-safety-only corpus).
  - mTCP: no analogous bucketing.
  - Ours `tools/packetdrill-shim-runner/src/classifier.rs:30-38`: new verdict with a soak-tested floor of 100 runs per script before pinning; gate accepts `exit < 128` (any non-signal exit).
  - **Resolution:** spec §1.1 G + §7 already cover the crash-safety-corpus discipline. The 100× soak floor is pinned via the `#[ignore] ligurio_no_crash_soak` test at `tools/packetdrill-shim-runner/tests/corpus_ligurio.rs` (added in T9 post-review fix via `LIGURIO_SOAK_ITERS` env var).

### FYI (informational — no action required)

- **I-1** — mTCP `tcp_util.c:13-59` `ParseTCPOptions` has an unfixed underflow: for an unknown kind it does `i += optlen - 2` without first asserting `optlen >= 2`, so a crafted `optlen = 0` or `optlen = 1` underflows the loop counter and walks off the segment. Our `crates/dpdk-net-core/src/tcp_options.rs` parser already documents and rejects this; nothing to action.

- **I-2** — mTCP `icmp.c` doesn't process `ICMP_FRAG_NEEDED` / update PMTU — RFC 1191 is unimplemented in their tree. Our `crates/dpdk-net-core/src/icmp.rs` covers it. The new ligurio `runnable-no-crash` bucket includes the malformed-ICMP scripts so we have a regression net for the path mTCP doesn't even have.

- **I-3** — mTCP `api.c::mtcp_write` (lines 1465-1572) doesn't NULL-check `buf` before forwarding to `CopyFromUser` (api.c:1423, called at api.c:1537); a caller passing `(NULL, len > 0)` would segfault inside the memcpy. Our `dpdk_net_send` rejects this with `-EINVAL`. Differentiator, not a finding.

- **I-4** — mTCP `tcp_in.c::ProcessRST` (line 196-252) carries an in-source TODO comment: "we need reset validation logic". RFC 9293 §3.10.7.4 (in-window check) is not implemented in mTCP. Our equivalent path validates and the new `probes/rst_ack.rs` tcpreq probes exercise it. The shivansh URG-vs-RST script (which we previously discussed flagging) processes RST regardless of URG, matching what mTCP does at tcp_in.c:1289-1296 — both behaviors are RFC-compliant; nothing to change.

- **I-5** — A8.5 introduces no new RACK / TLP / PAWS / RTT-estimator code, so there is nothing in the new diff for which "mTCP doesn't implement that" applies as a missing comparison. The phase is genuinely a test-discipline phase.
