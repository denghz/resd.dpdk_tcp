### Missing SHOULD (not in §6.4 allowlist)

- [x] **S-1** — RstAck probe under-tests its claim: scenario B (RST|URG) does not actually verify a flag-independent RST close. **RESOLVED (option 3):** added `AD-A8.5-rst-urg-precedence` deviation to master spec §6.4 (commit in T10 resolution). Updated `rst_ack.rs` probe scenario B to assert `+1 tcp.rx_rst` AND `+1 tcp.rx_urgent_dropped` (pins the documented deviation rather than claiming full independence). M5 matrix `Reset-Processing` row promoted PASS → PASS-WITH-DEVIATION citing the new AD row.
  - RFC clause: `docs/rfcs/rfc9293.txt:1509` — "In all states except SYN-SENT, all reset (RST) segments are validated by checking their SEQ fields. A reset is valid if its sequence number is in the window." `docs/rfcs/rfc9293.txt:1518` — "If the receiver was in any other state, it aborts the connection and advises the user and goes to the CLOSED state." (RFC 9293 §3.5.3 + §3.10.7.4)
  - Our code: `crates/dpdk-net-core/src/tcp_input.rs:730` drops URG-bearing segments at the *top* of `handle_established` — *before* the RST check at `:744`. A peer-sent `RST|URG` therefore returns `Outcome { tx: TxAction::None, urgent_dropped: true, .. }` with `closed: false` and `new_state: None`. The connection stays in ESTABLISHED. The engine still bumps `tcp.rx_rst` at `crates/dpdk-net-core/src/engine.rs:3617` (which fires *before* per-state dispatch on flag presence alone) and the new probe `tools/tcpreq-runner/src/probes/rst_ack.rs:88-94` only checks `tcp.rx_rst` deltas — so it passes despite no actual close happening.
  - Why not deferred: The probe is the only Layer-C citation for the M5 matrix row "Reset-Processing | RST processed independently of other flags". The M5 row is now misleading: it implies RST flag-independence is verified end-to-end, but only counter-presence is verified. A peer using `RST|URG` to abort a connection — the canonical real-world test of "independence from other flags" — will be silently dropped by our engine while the M5 matrix says we cover it. This is a coverage-claim accuracy bug, not a correctness bug (the silent-drop is consistent with `AD-A8-urg-dropped`); but it makes the M5 row aspirational rather than evidential.
  - Proposed fix (one of):
    1. **Strengthen probe.** Add `assert!(h.eng.conn_peer_mss(conn).is_some())` (i.e. the conn still exists / `accept_next` returned a valid handle) on scenario A (close should propagate `closed`); on scenario B, document that the URG-precedes-RST ordering means no close fires, and bump expectation from "+1 conn_rst" to "+0 conn_rst plus +1 rx_urgent_dropped" so the probe locks in the actual interaction. Then add a Layer-A `tests/rst_with_other_flags.rs` that exercises `RST|FIN`, `RST|PSH`, and bare `RST` on ESTABLISHED to show those *do* close (RST|URG is the documented exception under `AD-A8-urg-dropped`).
    2. **Reorder handler.** Move the RST check at `tcp_input.rs:744` *above* the URG check at `:730` so RST takes precedence per RFC 9293 §3.5.3. Costs: revisit `AD-A8-urg-dropped` text (the URG drop only applies to non-RST segments) and `cover_tcp_rx_urgent_dropped` may need adjustment if any test sends RST|URG.
    3. **Promote to §6.4 deviation.** Add a row `AD-A8.5-rst-urg-precedence` documenting that we drop `RST|URG` per the URG-first ordering (consequence of `AD-A8-urg-dropped`'s implementation choice). Update RstAck probe scenario B to assert the documented behavior (`+0 conn_rst, +1 rx_urgent_dropped`) so it pins the deviation rather than asserting behavior the engine doesn't implement.
  - The M5 matrix row "Reset-Processing | RST processed independently of other flags" should be edited in lockstep with whichever fix is chosen — currently it cites the new probe as PASS evidence, but the probe under-tests the claim.

### Accepted deviation (covered by spec §6.4)

- **AD-1** — `dpdk_net_shutdown(SHUT_RD | SHUT_WR)` returns `-EOPNOTSUPP`; only `SHUT_RDWR` performs the close
  - RFC clause: `docs/rfcs/rfc9293.txt:1645-1647` — "A host MAY implement a 'half-duplex' TCP close sequence, so that an application that has called CLOSE cannot continue to read data from the connection (MAY-1)." (RFC 9293 §3.6.1 — half-close is MAY-1, not a MUST.) `docs/rfcs/rfc9293.txt:3149-3153` — RFC 9293 §3.10.4 CLOSE Call ESTABLISHED STATE: "Queue this until all preceding SENDs have been segmentized, then form a FIN segment and send it. In any case, enter FIN-WAIT-1 state." Our `SHUT_RDWR` dispatch covers full-close per §3.10.4.
  - Spec §6.4 line: `docs/superpowers/specs/2026-04-17-dpdk-tcp-design.md:467` — "`AD-A8.5-shutdown-no-half-close` | POSIX `shutdown(2)` supports half-close via `SHUT_RD` / `SHUT_WR`, allowing the caller to close one direction while keeping the other open | `dpdk_net_shutdown(h, DPDK_NET_SHUT_RD)` and `dpdk_net_shutdown(h, DPDK_NET_SHUT_WR)` both return `-EOPNOTSUPP`. Only `DPDK_NET_SHUT_RDWR` performs the close (dispatching to `dpdk_net_close(h, 0)`). | Stage 1 byte-stream API has no half-close consumer. ... Promotion gate: HTTP/1.1 pipelining (Stage 3) or WebSocket close-frame handling (Stage 5) — both need half-close semantics — reopens this row."
  - Our code behavior: `crates/dpdk-net/src/lib.rs:1054-1058` — `match how { DPDK_NET_SHUT_RDWR => dpdk_net_close(p, conn, 0), DPDK_NET_SHUT_RD | DPDK_NET_SHUT_WR => -libc::EOPNOTSUPP, _ => -libc::EINVAL }`. RFC 9293 §3.6.1 half-close is `MAY-1` (line 1647), not a MUST, so opting out is RFC-compliant. SHLD-3 (line 1650) — "send a RST to show that data was lost" on CLOSE-with-pending-data — does not apply because we never enter half-close state. Promotion gate is well-defined. Tests: `crates/dpdk-net/tests/api_shutdown.rs` (4 tests).

- **AD-2** — `AD-A8-urg-dropped` interacts with RST|URG path
  - RFC clause: `docs/rfcs/rfc9293.txt:1518` — RFC 9293 §3.5.3: "If the receiver was in any other state, it aborts the connection and advises the user and goes to the CLOSED state." (Implies RST processing is unconditional after sequence validation.)
  - Spec §6.4 line: `docs/superpowers/specs/2026-04-17-dpdk-tcp-design.md:465` — `AD-A8-urg-dropped`: "URG-flagged inbound segments are dropped at the top of `established_rx` (`TxAction::None`, zero delivered bytes, no response segment) and `tcp.rx_urgent_dropped` is bumped."
  - Our code behavior: The URG drop check at `tcp_input.rs:730` runs before the RST check at `:744`. Strict reading: this is a *consequence* of `AD-A8-urg-dropped` (drop-on-URG is "always-on" per the §6.4 row text), but the row does not explicitly call out the RST|URG interaction. Finding S-1 above proposes either fixing the probe to pin this consequence, reordering the handler so RST takes precedence, or expanding `AD-A8-urg-dropped` to mention the RST|URG implication.

### FYI (informational — no action)

- **I-1** — MUST-2/3 (TCP checksum) zero-checksum exemption does not apply to TCP. RFC 9293 §3.1 (`docs/rfcs/rfc9293.txt:458-459`) is explicit: "The TCP checksum is never optional. The sender MUST generate it (MUST-2) and the receiver MUST check it (MUST-3)." Unlike UDP/IPv4 (RFC 768) where a transmitted zero csum means "no checksum present", TCP has no equivalent escape hatch. The probe's edge-case note about a 1-in-65536 randomly-chosen SYN folding to 0x0000 is correctly defensive — it documents that the canonical PEER_ISS = 0x1000 + fixed ports + fixed MSS produces a non-zero true checksum, so zeroing the field reliably triggers a mismatch. Implementation at `crates/dpdk-net-core/src/tcp_input.rs:130-134` reads the wire-stored csum, computes the expected csum over `[pseudo, head, zero_csum, tail]`, and returns `TcpParseError::Csum` on mismatch; engine then bumps `tcp.rx_bad_csum` at `engine.rs:3548`. PASS evidenced by the new probe.

- **I-2** — MUST-4 / MUST-6 / MUST-7 citations to §3.2 are correct. RFC 9293 §3.1 introduces the option machinery (`docs/rfcs/rfc9293.txt:495-523`) and §3.2 (`:532+`) provides the per-option formats. Both sections cross-reference each other; the M5 matrix cites §3.2 for all three MUSTs which is appropriate (the per-option detail lives in §3.2). Note the RFC text at `:521-523` describes MUST-7 with a *suggested* procedure of "reset the connection and log the error cause" — our LISTEN→passive-open path does `unwrap_or_default()` (tolerant fall-through) at `engine.rs:3587`, while our SYN_SENT→active-open path emits RST at `tcp_input.rs:609-619`. Both behaviors are MUST-7-compliant (the only MUST is "be prepared to handle ... without crashing"); the suggested-procedure RST is not normative. The probe correctly accepts either outcome at `tools/tcpreq-runner/src/probes/options.rs:272-365`.

- **I-3** — MUST-14 (MSS send + receive) is fully covered by the probe pair. RFC 9293 §3.7.1 (`docs/rfcs/rfc9293.txt:1734-1735`): "TCP endpoints MUST implement both sending and receiving the MSS Option (MUST-14)." `mss_support` (new, A8.5) walks the SYN-ACK option region TLV-by-TLV at `tools/tcpreq-runner/src/probes/mss.rs:377-460` to verify our outbound SYN-ACK carries an MSS option (kind=2, len=4) — covering the *sending* side. `missing_mss` (pre-existing, A8) verifies our parser correctly applies the 536 fallback when peer omits MSS — covering the *receiving* side. Together they pin both halves of MUST-14. No additional probe needed. Layer-A unit test `crates/dpdk-net-core/src/tcp_output.rs:411 syn_frame_has_mss_option_and_valid_sizes` provides redundant coverage of the emission half.

- **I-4** — TX option-order alignment is RFC-compliant. RFC 9293 §3.2 (`docs/rfcs/rfc9293.txt:514-523`) is explicit that receivers MUST handle options in any order: "A TCP implementation MUST be able to receive a TCP Option in any segment (MUST-5)" and unknown options "MUST ignore without error" (MUST-6). The new TX emission order `<MSS, NOP+WSCALE, [SACKP|SACKP+TS|NOPs+TS], [NOPs+SACK-blocks]>` at `crates/dpdk-net-core/src/tcp_options.rs:145-225` matches Linux's `tcp_options_write` for corpus-compatibility. The receiver-side parser at `tcp_options.rs:247-356` is a per-byte TLV walk indexed by `kind` — it has no order dependency. The "parser remains order-agnostic" claim is accurate and is exercised by the existing unit tests at `tcp_options.rs:386-416 ack_with_timestamp_and_two_sack_blocks_word_aligned` and the round-trip test at `:470 parse_full_syn_options_round_trips_encode`.

- **I-5** — `dpdk_net_shutdown(SHUT_RDWR)` correctly maps to RFC 9293 §3.10.4 CLOSE Call. `docs/rfcs/rfc9293.txt:3149-3153` specifies the ESTABLISHED CLOSE behavior (form FIN, enter FIN-WAIT-1). `crates/dpdk-net/src/lib.rs:1055` dispatches `SHUT_RDWR` to `dpdk_net_close(p, conn, 0)` which is the existing FIN-handshake path — already exercised end-to-end by `crates/dpdk-net-core/tests/test_server_active_close.rs`. The 4 new tests at `crates/dpdk-net/tests/api_shutdown.rs` cover the dispatcher arms (SHUT_RD / SHUT_WR / invalid-how / SHUT_RDWR-vs-close-equivalence) without re-exercising the core-crate active-close path. The `closed` field of `outcome` and `tcp.conn_close` counter are bumped by the underlying `dpdk_net_close` path; no separate Shutdown-API counter is needed.

- **I-6** — The new M5 matrix row `Shutdown-API` correctly cites Layer-A end-to-end coverage (`tests/test_server_active_close.rs`) plus the new Layer-A dispatcher tests (`tests/api_shutdown.rs`). Status `DEVIATION — AD-A8.5-shutdown-no-half-close` is correctly classified: full-close is RFC-default exact (PASS-equivalent), half-close is the explicit deviation. Cross-reference at `docs/superpowers/reports/stage1-rfc793bis-must-matrix.md:38` and §6.4 line 467 are aligned.

- **I-7** — The G workstream (crash-safety corpus tests) does not introduce new RFC-clause coverage claims. Per A8.5 spec §1.1 G, the 6 corpus scripts (icmp-all-types, 3 syscall_bad_arg, 2 PMTU) verify engine robustness against unexpected peer behavior; they exercise existing counters (`ip.rx_icmp`, `ip.rx_icmp_frag_needed`, `ip.pmtud_updates`, `tcp.rx_bad_option`, `tcp.rx_bad_csum`) and existing API EINVAL/EOPNOTSUPP returns. RFC 1191 (PMTUD) and RFC 792 (ICMP) compliance was already pinned by the earlier phases that landed those features; the corpus scripts add stability evidence (no SIGSEGV/SIGABRT under 600 iterations) rather than new clause coverage. No additional findings here.

- **I-8** — The Workstream C google env-stubs (`0007-google-env-stubs.patch`) is a build-environment fix, not an RFC behavior change. It replaces the upstream `defaults.sh` (which calls `sysctl` and writes to `/proc/sys/net/...`) and `set_sysctls.py` (which mutates kernel sysctls) with no-op stubs so the corpus scripts do not exit 127 at the shell-init step. This is reviewability-equivalent to a runtime fixture but lives in the patch tree per spec §3.5. No RFC clause is touched.
