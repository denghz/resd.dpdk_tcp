## Verdict

PROCEED-WITH-FIXES. The phase landed a small, modular, well-tested crate with good separation of pure-data scenario matrix, pure-function FSM oracle, and engine-driven runner. However, the post-merge `8147404` "gate behind test-server feature" fix introduced a critical orchestrator-script regression ŌĆö `scripts/layer-h-{smoke,nightly}.sh` now build the workspace WITHOUT `--features test-server`, which (combined with `required-features = ["test-server"]` and `default = []`) means the binary is NEVER built and both scripts will fail at the post-build existence check on master HEAD. Additionally, a non-blocking but real correctness gap on row 14 (the disjunctive corruption assertion omits `tcp.rx_bad_csum`, the only counter that fires under offload-OFF for TCP-payload corruption) and a confirmed always-pass property of the PR #9 RX-leak side-checks under the smoke workload's intensity reduce the gate's effective coverage below what the spec advertises.

## Architectural drift

- **AD-1 ŌĆö Orchestrator scripts don't pass `--features test-server`; both layer-h scripts are broken at HEAD.** `tools/layer-h-correctness/Cargo.toml:22` carries `required-features = ["test-server"]` on the bin target and `default = []` at line 36; the lib at `tools/layer-h-correctness/src/lib.rs:17` is `#![cfg(feature = "test-server")]`. Commit `8147404` introduced this gating but did NOT update `scripts/layer-h-smoke.sh:94` or `scripts/layer-h-nightly.sh:101`, both of which run `timeout 600s cargo build --release --workspace` with no feature flag. Cargo silently skips bins whose `required-features` are unmet, so `target/release/layer-h-correctness` will not exist and the post-build existence check (`scripts/layer-h-smoke.sh:233-238`, `scripts/layer-h-nightly.sh:240-245`) fails with `missing after build`. **This is the same class of bug `8147404` was meant to fix on a different vector** ŌĆö there is no CI gate, smoke run, or stage-cut nightly that exercises layer-h-correctness on master HEAD; the regression has been silently in place since `8147404` (2026-05-04). Fix shape: add `--features test-server` to the `cargo build` lines, and either add `-p layer-h-correctness` so feature-unification is bin-scoped or add a follow-up `cargo build -p layer-h-correctness --release --features test-server` step after the workspace build. Recommend the latter ŌĆö `--features test-server` on `cargo build --workspace` reactivates the original gateway-ARP regression class for every other bench-* binary in the workspace build cache.
- **AD-2 ŌĆö `Engine::state_of` is unconditionally public; the test-server gating rationale in `lib.rs:7` is stale.** `tools/layer-h-correctness/src/lib.rs:7` justifies `#![cfg(feature = "test-server")]` with "layer-h-correctness's source uses Engine::state_of (test-server only)." But `state_of` at `crates/dpdk-net-core/src/engine.rs:6659` carries no `#[cfg]` and is unconditionally public. The comment at `engine.rs:2451` ("test-server-gated `state_of`") is also stale. The actual reason layer-h needs `test-server` is unclear from inspection ŌĆö none of the engine API calls (`workload.rs:111,114,149,159,182`, `observation.rs:646,660,681`) require the feature. This isn't a correctness bug, but it suggests the `test-server` feature flag was conservatively chosen as the umbrella for the regression-fix's optionality contract rather than because it's strictly needed; a future maintainer reading the comment will be misled about what they can/can't unbundle.
- **AD-3 ŌĆö `bench-e2e` and `bench-stress` are pulled in via the `test-server` feature but neither has a `test-server` feature themselves.** `tools/layer-h-correctness/Cargo.toml:42-48` defines `test-server = ["dep:bench-e2e", "dep:bench-stress", ...]`. Verified at `tools/bench-e2e/Cargo.toml` (no `[features]` section) and `tools/bench-stress/Cargo.toml` (similarly). The Cargo.toml works fine because the `dep:` directives only enable the optional dependency, not a feature on it ŌĆö but the naming "test-server" feature including non-test-server crates is misleading. Future-foot-gun if someone adds a real `test-server` feature to `bench-stress`.

## Cross-phase invariant violations

- **CPI-1 ŌĆö gateway-ARP regression class only HALF fixed; `scapy-fuzz-runner` still has the same architectural pattern at HEAD.** Confirmed by reading `tools/scapy-fuzz-runner/Cargo.toml:8`:
  ```
  dpdk-net-core = { path = "../../crates/dpdk-net-core", features = ["test-inject"] }
  ```
  `dpdk-net-core` is non-optional and `test-inject` is requested unconditionally. Cargo's feature unification activates `test-inject` on `dpdk-net-core` for ALL workspace consumers when running `cargo build --workspace --release`. `test-inject` does NOT reroute `tx_frame` (so the gateway-ARP symptom doesn't reproduce), but the divergent `inject_rx_frame` implementation at `crates/dpdk-net-core/src/engine.rs:6266` (under `#[cfg(feature = "test-inject")]`) gets pulled into every workspace consumer's build. Recommend: same fix shape that `8147404` applied to layer-h-correctness ŌĆö `optional = true` on the `dpdk-net-core` dep and a non-default `test-inject` feature gate. **This was already independently flagged as Part 6 AD-1 / S2-1.** Re-flagging because Part 9's review-focus list explicitly asked to confirm.
- **CPI-2 ŌĆö There is no CI / metadata gate on workspace feature unification across runner crates.** Both T11 (`f6280ab`) and T20 (`8147404`) regressions made it into master because workspace feature resolution is not validated in CI. Recommend a CI step (1-line bash + `cargo metadata`) that asserts `dpdk-net-core`'s resolved feature set on a `cargo build --workspace --release` is exactly `default` plus nothing else. Would have caught both T11 and T20 at PR-time. Already implicit in Part 6 S2-1; promoting here because Part 9's bin existence regression (AD-1) has the same root cause-class: no script-side validation.

## Tech debt accumulated

- **TD-1 ŌĆö `EventKind::Other` fallthrough loses event timestamps.** `tools/layer-h-correctness/src/observation.rs:240-249` emits `emitted_ts_ns: 0` for unhandled `InternalEvent` variants (notably `Readable`, `Writable`, `ApiTimer`). The fallthrough is documented as deliberate, but the `Other` records in failure bundles are forensically near-useless because they carry neither timestamp nor identifying data. Already noted in `phase-a10-5-mtcp-compare.md` I-6; not blocking. Low-cost fix: extract `emitted_ts_ns` from any `InternalEvent` variant via a helper method on the source enum (would require a one-line addition in `crates/dpdk-net-core/src/tcp_events.rs`, but A10.5's "no engine-side changes" promise leaves this for a follow-up).

That's a forward-looking promise; not strictly debt, but if Stage 2 doesn't migrate `ConnHandle`, the allow becomes accidental tech debt. Acceptable as-is; flagging because it's load-bearing on a future-tense promise.
- **TD-3 ŌĆö Two helpers each declare `#[allow(clippy::too_many_arguments)]`.** `report.rs:203` (`write_failure_bundle` ŌĆö 9 args), `workload.rs:54` (`run_one_scenario` ŌĆö 8 args). Both are organic API growth; both could be refactored into a small `Args` struct, but the readability win is marginal. Acceptable for a tooling crate. Flagging for completeness.

## Test-pyramid concerns

- **TP-1 ŌĆö A10.5 covers ONLY netem-stress correctness, not max-throughput correctness.** User noted "many bugs especially for max throughput bench"; the spec at `2026-05-01-stage1-phase-a10-5-layer-h-correctness-design.md:13` is explicit that "performance / latency under netem ŌĆö owned by A10's bench-stress". A10.5 has 30 s ├Ś 17 scenarios ├Ś 1 connection ├Ś 128 B request/response. Max-throughput (T20-T23 territory) operates at 100k+ conn/sec / 1+ Gbps with stalls + retransmit storms; A10.5 will NEVER exercise those code paths. The gate's name ("Layer H correctness") oversells coverage ŌĆö it's "Layer H WAN-adversity correctness at low throughput". Recommend: a parallel A10.5b phase (later) that runs the same 17-scenario matrix at max-throughput intensities OR add explicit non-coverage language to spec ┬¦1 / ┬¦2 stating "max-throughput correctness is not covered by this phase". The stalls T21/T22/T23 surfaced (drop-site counter, mTCP driver pump, etc.) are exactly the sort of bug A10.5 would NEVER catch.
- **TP-2 ŌĆö RX-mempool-avail side-check is always-pass under smoke workload intensity.** Confirmed by inspection: `tcp.rx_mempool_avail` is sampled once per second (`engine.rs:2515-2535`, gated on `now_tsc.wrapping_sub(last) >= tsc_hz`). The smoke set runs ~1 ms observation batches at single-connection 128 B RTT under 50 ms netem delay ŌåÆ effective RX rate ~20 pps. Default RX mempool is in the thousands of mbufs. The level never approaches the 32-mbuf floor; the side-check at `observation.rs:681-684` is a no-op under smoke. Spec ┬¦4 / ┬¦5.2 frames this side-check as a meaningful PR #9 leak-detect gate, but it's only meaningful at A10's bench-stress intensities (which A10.5 doesn't reach). Recommend: either (a) document the gate as "tripwire for nightly only ŌĆö smoke intensity will never trigger it", (b) raise the floor under smoke to a value that exercises the read path (still not the leak-detect logic, but at least proves the read works), or (c) replace with `tcp.mbuf_refcnt_drop_unexpected delta == 0` as the only side-check (which IS load-independent because it fires on Drop with refcount > 8 ŌĆö a logic error, not a level threshold). The `mbuf_refcnt_drop_unexpected` end-of-scenario check at `assertions.rs:202-226` IS meaningful regardless of intensity.
- **TP-3 ŌĆö Disjunctive corruption assertion (row 14) only exercises offload-on path.** `dpdk-net-core`'s default features include `hw-offload-rx-cksum` (`crates/dpdk-net-core/Cargo.toml:20`). Layer-h orchestrator scripts pass nothing to `cargo build` (`scripts/layer-h-smoke.sh:94`, `scripts/layer-h-nightly.sh:101`), so they always build with offload-on. The offload-off path of the disjunctive (`ip.rx_csum_bad`) is never exercised by any orchestrator script in the repo. Spec ┬¦4 row 14 sells this as "passes under either build profile"; in practice only one profile is ever built. Recommend: either drop the offload-off arm of the disjunctive (admit the gate is offload-on-only) or add an offload-off variant to the nightly's invocation matrix. See also TP-4 for a further correctness issue with the offload-off path.
- **TP-4 ŌĆö Disjunctive corruption assertion misses `tcp.rx_bad_csum`; would spuriously fail under offload-off if exercised.** `tools/layer-h-correctness/src/scenarios.rs:235-238` defines the disjunctive group as `[eth.rx_drop_cksum_bad, ip.rx_csum_bad]`. With offload OFF and `corrupt 0.01%`, netem flips one byte per ~10kB. Most corrupted bytes land in TCP payload (the IP header is only 20 bytes vs hundreds of payload bytes per segment). Software IP-cksum verify (`crates/dpdk-net-core/src/l3_ip.rs:104-114`) only covers the IP HEADER ŌĆö payload corruption passes IP-cksum cleanly. The TCP-cksum software verify (`engine.rs:4068`) bumps `tcp.rx_bad_csum`, NOT `ip.rx_csum_bad`. So under offload-off the disjunctive `[eth.rx_drop_cksum_bad, ip.rx_csum_bad]` will spuriously FAIL because neither counter fires for TCP-payload corruption. Fix: add `tcp.rx_bad_csum` to the disjunctive group: `&["eth.rx_drop_cksum_bad", "ip.rx_csum_bad", "tcp.rx_bad_csum"]`. Note this is not currently caught because TP-3 ŌĆö the offload-off path is never built/run.
- **TP-5 ŌĆö `rx_rst == 0` on row 14 is anchored to `tcp.rx_rst`, but the smoke workload doesn't generate RSTs even when adversity is real.** `scenarios.rs:233` has `("tcp.rx_rst", "==0")`. This expects the peer never to send RST. Under the bench-pair fleet with peer running `echo-server`, the peer doesn't send RST on cksum errors ŌĆö corrupted segments are just dropped at NIC/L3, no reaction. So this assertion passes regardless of whether the corruption ever happens. It's a defensive-coding-style gate but never actually exercises the engine's RST-handling. Acceptable as-is; flagging because it's a counter assertion that won't catch the failure mode it nominally guards.

## Observability gaps

- **OG-1 ŌĆö `tcp.mbuf_refcnt_drop_unexpected` is the only side-check that's actually meaningful at smoke intensity.** Already covered under TP-2; flagging here because the spec ┬¦4 "Global side-checks" section sells three side-checks as if they're equivalently load-bearing. Only one (`mbuf_refcnt_drop_unexpected delta == 0`) is intensity-independent.
- **OG-2 ŌĆö `obs.events_dropped` per-batch delta side-check is a fire alarm without a backing observable.** `observation.rs:686-693` checks `obs.events_dropped` delta per batch. The engine's bounded event queue has a 1024-entry capacity (per `crates/dpdk-net-core/src/tcp_events.rs:119` ŌĆö default `VecDeque`); a single connection's event rate is < 100/s so the queue never overflows under smoke. End-of-scenario assertion at `counter_expectations` for `obs.events_dropped == 0` covers the same ground. Per-batch + per-scenario form is redundant under smoke. Acceptable defense-in-depth.
- **OG-3 ŌĆö `EventKind::Other` swallows `Readable` / `Writable` / `ApiTimer` events with no per-kind counter exposure for the bundle.** The bundle records only kind, not subkind; under `corruption_001pct`, a partial-read split could fire a `Readable` event that's recorded as `Other` with `emitted_ts_ns=0`. See TD-1.

## Memory-ordering / ARM-portability concerns

None. All atomic accesses in `tools/layer-h-correctness/src/` use `Ordering::Relaxed` for level reads (`observation.rs:681,687`, `counters_snapshot.rs:46,86`), which is safe across ARM/x86. No explicit `asm!`, `core::arch`, or `target_arch` references in the crate. No memory-barrier primitives. The `EventRing` is `VecDeque<EventRecord>` ŌĆö no atomics, single-threaded. No portability concerns.
