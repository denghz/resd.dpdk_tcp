# Part 4 Cross-Phase Retro Review (Codex)
**Reviewer:** codex:codex-rescue
**Reviewed at:** 2026-05-05
**Part:** 4 — Public API + zero-copy + FFI safety
**Phases:** A6, A6.5, A6.6, A6.7

## Verdict
[BUG] HEAD still has two public API / zero-copy correctness defects in the A6.6 surface: `dpdk_net_poll` invalidates queued READABLE event backing before it may drain older events (`crates/dpdk-net/src/lib.rs:504`, `crates/dpdk-net-core/src/engine.rs:2554`, `crates/dpdk-net-core/src/engine.rs:3681`), and the real RX hot path parses a chained mbuf through the head segment only before the chain walker can run (`crates/dpdk-net-core/src/lib.rs:74`, `crates/dpdk-net-core/src/engine.rs:3756`, `crates/dpdk-net-core/src/l3_ip.rs:85`).

[BUG] The in-tree FFI smoke test is currently an ABI drift example rather than a guard: its hand-written `Cfg` mirror stops before the public `rx_mempool_size` tail field (`tests/ffi-test/tests/ffi_smoke.rs:63`, `tests/ffi-test/tests/ffi_smoke.rs:106`), while the generated header and Rust ABI include that field (`include/dpdk_net.h:128`, `crates/dpdk-net/src/api.rs:67`) and `dpdk_net_engine_create` reads it (`crates/dpdk-net/src/lib.rs:213`).

[FYI] The PR #9-class mbuf leak fix is intact at HEAD: `MbufHandle::Drop` now pre-reads the refcount and releases via `shim_rte_pktmbuf_free_seg` instead of a bare refcount decrement (`crates/dpdk-net-core/src/mempool.rs:261`, `crates/dpdk-net-core/src/mempool.rs:273`), and `Engine::drop` clears flow-table and queued mbuf owners before mempools are dropped (`crates/dpdk-net-core/src/engine.rs:6541`, `crates/dpdk-net-core/src/engine.rs:6556`).

## Architectural drift
[FYI] The requested core review targets `api.rs`, `events_queue.rs`, and `mbuf_handle.rs` are not literal files at HEAD; the reviewed equivalents are `EventQueue` in `tcp_events.rs` (`crates/dpdk-net-core/src/tcp_events.rs:119`), `MbufHandle` in `mempool.rs` (`crates/dpdk-net-core/src/mempool.rs:204`), and the unsafe mbuf slice helper in core `lib.rs` (`crates/dpdk-net-core/src/lib.rs:74`).

[SMELL] `Engine::events()` still exposes a mutable `RefMut<EventQueue>` from the core engine (`crates/dpdk-net-core/src/engine.rs:2483`) even though the C ABI normally translates through `drain_events` (`crates/dpdk-net-core/src/engine.rs:3681`). That keeps the event queue as a mutable internal structure rather than a typed event-delivery boundary, which matters because READABLE events reference per-connection scratch indirectly (`crates/dpdk-net-core/src/tcp_events.rs:46`).

## Cross-phase invariant violations
[BUG] READABLE event lifetime is broken when events remain queued across C `dpdk_net_poll` calls. The C entry point calls `e.poll_once()` before it checks `events_out` / `max_events` or drains anything (`crates/dpdk-net/src/lib.rs:504`, `crates/dpdk-net/src/lib.rs:505`); top-of-poll clears every connection's `delivered_segments` and `readable_scratch_iovecs` (`crates/dpdk-net-core/src/engine.rs:2554`, `crates/dpdk-net-core/src/engine.rs:2570`); and `drain_events` drains only up to the caller's `max` (`crates/dpdk-net-core/src/engine.rs:3681`, `crates/dpdk-net-core/src/engine.rs:3683`). A READABLE event stores only `conn`, `seg_idx_start`, `seg_count`, and `total_len` (`crates/dpdk-net-core/src/tcp_events.rs:46`, `crates/dpdk-net-core/src/tcp_events.rs:52`), so a later drain reconstructs `segs` from the now-cleared scratch vector (`crates/dpdk-net/src/lib.rs:530`, `crates/dpdk-net/src/lib.rs:543`) and may return a dangling or empty view to C. This violates the header promise that the array and mbuf-backed bases remain valid until the next poll, because the queued event can be invalidated before it is ever delivered (`include/dpdk_net.h:164`).

[BUG] The chained-mbuf RX zero-copy path still reaches L3 with a head-only byte slice. `dispatch_one_real_mbuf` calls `mbuf_data_slice(m)` and then `rx_frame(bytes, ...)` (`crates/dpdk-net-core/src/engine.rs:3756`, `crates/dpdk-net-core/src/engine.rs:3790`), while `mbuf_data_slice` uses only `shim_rte_pktmbuf_data_len(m)` for the current mbuf segment (`crates/dpdk-net-core/src/lib.rs:74`, `crates/dpdk-net-core/src/lib.rs:76`). The IPv4 parser rejects a packet when `total_len > pkt.len()` (`crates/dpdk-net-core/src/l3_ip.rs:85`, `crates/dpdk-net-core/src/l3_ip.rs:87`), and the engine maps that to `ip.rx_drop_short` (`crates/dpdk-net-core/src/engine.rs:3924`). The later TCP chain walker exists (`crates/dpdk-net-core/src/tcp_input.rs:1128`, `crates/dpdk-net-core/src/tcp_input.rs:1176`), but it runs only after L2/L3/TCP parsing has already accepted the head slice, so a legitimate multi-segment frame with IP total length spanning the chain is dropped before the walker can preserve the tail segments.

## Tech debt accumulated
[SMELL] The READABLE translation path relies on comments rather than an enforced owner type. The C ABI comment says scratch is not mutated until the app has consumed the event (`crates/dpdk-net/src/lib.rs:510`, `crates/dpdk-net/src/lib.rs:514`), but the actual control flow can run a new `poll_once` before a previously queued READABLE event is drained (`crates/dpdk-net/src/lib.rs:504`, `crates/dpdk-net-core/src/engine.rs:3681`). This is a mechanical debt item because `InternalEvent::Readable` carries indices, not owned or copied iovec data (`crates/dpdk-net-core/src/tcp_events.rs:46`, `crates/dpdk-net-core/src/tcp_events.rs:52`).

[SMELL] `EventQueue` overflow behavior is implemented as drop-oldest with relaxed counter accounting (`crates/dpdk-net-core/src/tcp_events.rs:153`, `crates/dpdk-net-core/src/tcp_events.rs:168`), but the local unit tests only cover FIFO and outstanding length (`crates/dpdk-net-core/src/tcp_events.rs:208`, `crates/dpdk-net-core/src/tcp_events.rs:237`). There is no same-module assertion that overflow drops exactly one old event, bumps `obs.events_dropped`, and latches high-water.

## Test-pyramid concerns
[BUG] `tests/ffi-test/tests/ffi_smoke.rs` hand-mirrors `dpdk_net_engine_config_t` and omits `rx_mempool_size` (`tests/ffi-test/tests/ffi_smoke.rs:63`, `tests/ffi-test/tests/ffi_smoke.rs:106`), then casts the shorter Rust stack object to the public config pointer (`tests/ffi-test/tests/ffi_smoke.rs:143`). The production ABI includes the tail field (`crates/dpdk-net/src/api.rs:67`, `include/dpdk_net.h:128`) and reads it during create (`crates/dpdk-net/src/lib.rs:213`), so the smoke test can cause an out-of-bounds read instead of catching C header drift.

[SMELL] The existing multi-segment zero-copy test validates the TCP chain walker but not the real RX parser path that fails. The test explicitly direct-constructs a `TcpConn` and calls `tcp_input::dispatch` with a `ParsedSegment` over the head link (`crates/dpdk-net-core/tests/rx_zero_copy_multi_seg.rs:12`, `crates/dpdk-net-core/tests/rx_zero_copy_multi_seg.rs:159`), while production dispatch enters through `dispatch_one_real_mbuf -> rx_frame` (`crates/dpdk-net-core/src/engine.rs:3756`, `crates/dpdk-net-core/src/engine.rs:3790`). That leaves the L2/L3 total-length rejection path untested.

[SMELL] The C ABI smoke test comment says its byte shim must match the cbindgen layout (`tests/ffi-test/tests/ffi_smoke.rs:63`), but it does not include or bindgen the generated header struct that actually contains `rx_mempool_size` (`include/dpdk_net.h:128`). This turns the test into a second schema to maintain by hand.

## Observability gaps
[BUG] The `test-inject` multi-segment RX path misses the packet counter bump that the single-frame injection path explicitly added. Normal polling bumps `eth.rx_pkts` per received burst (`crates/dpdk-net-core/src/engine.rs:2609`, `crates/dpdk-net-core/src/engine.rs:2610`), `inject_rx_frame` mirrors that with a direct `rx_pkts` increment (`crates/dpdk-net-core/src/engine.rs:6352`, `crates/dpdk-net-core/src/engine.rs:6362`), and `dispatch_one_real_mbuf` only adds bytes (`crates/dpdk-net-core/src/engine.rs:3783`, `crates/dpdk-net-core/src/engine.rs:3784`). `inject_rx_chain` allocates, links, and dispatches the head (`crates/dpdk-net-core/src/engine.rs:6389`, `crates/dpdk-net-core/src/engine.rs:6492`) without an equivalent `eth.rx_pkts` increment in that path, so chain injection under-reports RX packet counters.

[SMELL] The stale READABLE scratch failure has no observable error path at the ABI boundary. If `flow_table().get(conn)` still returns the connection, `dpdk_net_poll` emits a READABLE event using `readable_scratch_iovecs.as_ptr().add(seg_idx_start)` (`crates/dpdk-net/src/lib.rs:530`, `crates/dpdk-net/src/lib.rs:543`) and does not validate that the current scratch length still covers `seg_count` (`crates/dpdk-net/src/lib.rs:549`, `crates/dpdk-net/src/lib.rs:551`).

## Memory-ordering / ARM-portability concerns
[FYI] The old SPSC ring-index / wrap concern is not present in the current event queue implementation. `EventQueue` is a `VecDeque<InternalEvent>` behind the engine's `RefCell` (`crates/dpdk-net-core/src/tcp_events.rs:119`, `crates/dpdk-net-core/src/engine.rs:708`), and producer / consumer operations are plain `push_back` / `pop_front` under mutable borrow (`crates/dpdk-net-core/src/tcp_events.rs:161`, `crates/dpdk-net-core/src/tcp_events.rs:181`). The only atomics in that queue path are observability counters using `Relaxed` (`crates/dpdk-net-core/src/tcp_events.rs:170`, `crates/dpdk-net-core/src/tcp_events.rs:177`).

[FYI] Relaxed counter loads/stores are explicitly documented as snapshots, not synchronization. Core counters state hot-path increments and cross-lcore snapshots use `Relaxed` (`crates/dpdk-net-core/src/counters.rs:3`, `crates/dpdk-net-core/src/counters.rs:5`), and the public C ABI tells C/C++ readers to use the atomic-load helper, especially on ARM (`crates/dpdk-net/src/api.rs:295`, `crates/dpdk-net/src/api.rs:299`, `include/dpdk_net.h:231`, `include/dpdk_net.h:236`).

[FYI] The mbuf refcount path is not a Rust atomic ordering pair. Ownership is transferred through DPDK refcount primitives: `MbufHandle::try_clone` bumps immediately before constructing a new handle (`crates/dpdk-net-core/src/mempool.rs:250`, `crates/dpdk-net-core/src/mempool.rs:254`), and `Drop` releases through `shim_rte_pktmbuf_free_seg` (`crates/dpdk-net-core/src/mempool.rs:261`, `crates/dpdk-net-core/src/mempool.rs:283`). I did not find a Relaxed producer/consumer mbuf handoff that would become invalid only on ARM.

## C-ABI / FFI
[BUG] The C ABI config tail field is present in both Rust and the generated header (`crates/dpdk-net/src/api.rs:67`, `include/dpdk_net.h:128`), but the in-tree FFI smoke mirror lacks it (`tests/ffi-test/tests/ffi_smoke.rs:63`, `tests/ffi-test/tests/ffi_smoke.rs:106`). Since `dpdk_net_engine_create` immediately binds `let cfg = &*cfg` and later reads `cfg.rx_mempool_size` (`crates/dpdk-net/src/lib.rs:148`, `crates/dpdk-net/src/lib.rs:213`), the stale mirror is a concrete unsafe FFI precondition violation.

[SMELL] `dpdk_net_engine_create` has no ABI size or version guard around `dpdk_net_engine_config_t`; it checks only for a null pointer and then treats the pointer as the current full struct (`crates/dpdk-net/src/lib.rs:141`, `crates/dpdk-net/src/lib.rs:148`). That makes tail-field additions depend on every caller and test being rebuilt against the latest generated header (`include/dpdk_net.h:78`, `include/dpdk_net.h:139`).

## Hidden coupling
[BUG] `drain_events` invokes the C ABI closure with `&Engine` after popping the event (`crates/dpdk-net-core/src/engine.rs:3681`, `crates/dpdk-net-core/src/engine.rs:3687`), and the closure re-enters the engine to borrow the flow table for READABLE view reconstruction (`crates/dpdk-net/src/lib.rs:509`, `crates/dpdk-net/src/lib.rs:530`). That is mechanically coupled to the top-of-`poll_once` scratch clearing site (`crates/dpdk-net-core/src/engine.rs:2554`, `crates/dpdk-net-core/src/engine.rs:2571`): if the event is not drained in the same ABI call that produced it, the closure translates stale indices against fresh or empty scratch.

[FYI] I did not find a RefCell re-entrancy panic in `drain_events` itself: the event-queue borrow is scoped to the `pop()` expression (`crates/dpdk-net-core/src/engine.rs:3684`) before the sink runs (`crates/dpdk-net-core/src/engine.rs:3687`). The defect is lifetime/order coupling, not an active nested borrow of the same `RefCell`.

## Documentation drift
[SMELL] RX mempool formula documentation is split between the updated runtime/getter formula and stale public config-field comments. Core runtime computes `4 * max_connections * per_conn + 4096` (`crates/dpdk-net-core/src/engine.rs:1186`, `crates/dpdk-net-core/src/engine.rs:1191`), and the getter docs say the same (`crates/dpdk-net/src/lib.rs:603`, `crates/dpdk-net/src/lib.rs:609`, `include/dpdk_net.h:575`, `include/dpdk_net.h:581`). The config struct field doc in Rust and the generated header still say `2 * max_connections * ceil(...) + 4096` (`crates/dpdk-net/src/api.rs:67`, `crates/dpdk-net/src/api.rs:70`, `include/dpdk_net.h:128`, `include/dpdk_net.h:132`).

[SMELL] The `dpdk_net_poll` READABLE comment says top-of-next-`poll_once` runs after the app has consumed the event (`crates/dpdk-net/src/lib.rs:514`, `crates/dpdk-net/src/lib.rs:516`), but the function can return without draining any events when `events_out` is null or `max_events == 0` after already calling `poll_once` (`crates/dpdk-net/src/lib.rs:504`, `crates/dpdk-net/src/lib.rs:505`). The comment should not be treated as an invariant proof.
