## Verdict

**SHIP** — no blocking architectural defects. The A5/A5.5/A5.6 modules are coherent, the `tcp_per_packet_events` runtime gate is uniformly applied at every per-packet emit site (RTO/TLP/RACK), timer-cancel discipline is preserved across all three teardown paths (force-close, transition→Closed, reap_time_wait), and the bug_008 RACK u64-ns migration was sound. Findings are accumulated debt and documentation drift: a public-ABI doc-comment that promised the cbindgen-emitted header would carry the wraparound + threading contract but does not (only ~3 sentences shipped); a self-acknowledged spec-vs-code mismatch in `select_bucket` whose §7.1 expectation was never reconciled with the §3.1 ladder rule; and one mempool-teardown comment that overstates what `Drop for Engine` actually does for `snd_retrans` `Mbuf` (non-owning) handles. None of these break A6/A6.5 invariants from A5/A5.5.

## Architectural drift

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_rack.rs:107-117` carries a kept-for-tests `rack_mark_losses_on_rto` wrapper that allocates a fresh `Vec<u16>` per call.** A6.5 Task 10 added the alloc-free `_into` variant but the wrapper was kept "for tests and any non-hot-path caller that prefers the owning-Vec ergonomics" (paraphrasing the sibling `prune_below` rationale at `tcp_retrans.rs:97-100`). The engine uses only the `_into` form (`engine.rs:2980`), so the wrapper is now exclusively reachable from the in-file unit tests. That's fine but it means the alloc-free property is tested only indirectly — every change to the inner helper has to manually update both forks, and the wrapper exists purely to avoid touching tests.

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_retrans.rs:101-141` has the same dual-API shape** (`prune_below` returning `SmallVec<[RetransEntry; 8]>` vs `prune_below_into_mbufs` writing into engine scratch). Engine uses `_into_mbufs` exclusively for the hot path; `prune_below` is only reachable from in-file tests. Same fork-maintenance hazard.

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_events.rs:148` — `EventQueue::with_cap` caps initial heap capacity at `DEFAULT_SOFT_CAP=4096` even when the caller requests `soft_cap = 64K+`.** Any soft-cap above 4096 incurs amortized VecDeque growth during the ramp from 4096 to the cap; this contradicts the Stage-1 audit's "pre-size to high-water mark" pattern (`tcp_conn.rs:420 timer_ids cap=32`, `tcp_conn.rs:430 snd_retrans pre-sized to send-buf ceiling`, `tcp_timer_wheel.rs:90 BUCKET_INIT_CAP=512`). Either grow on first overflow (current behavior) or pre-size to `cap` always; the half-measure of capping at 4096 is an arbitrary line nothing else in the codebase respects.

## Cross-phase invariant violations

- **None at the per-packet event gate.** Every `InternalEvent::TcpRetrans` and `InternalEvent::TcpLossDetected` push site (`engine.rs:3045-3093` RTO, `engine.rs:3236-3263` TLP, `engine.rs:4292-4321` RACK) is gated by `if self.cfg.tcp_per_packet_events` AND `#[cfg(not(feature = "obs-none"))]`. No A6/A6.5 emit site bypasses the runtime gate. `InternalEvent::Writable` (`engine.rs:4256-4266`) and `InternalEvent::ApiTimer` (`engine.rs:2887-2905`) intentionally bypass `tcp_per_packet_events` — they're public-API state events, not per-packet observability — which matches the A5 design spec §9.2 and A6 design spec §3.3 contracts.

- **Timer-cancel discipline survives every teardown path.** `force_close_etimedout` (`engine.rs:3498-3534`), the `outcome.closed`-driven removal (`engine.rs:4626-4652`), and `reap_time_wait` (`engine.rs:3649-3675`) all collect `conn.timer_ids` plus the three named handles (`rto_timer_id`, `tlp_timer_id`, `syn_retrans_timer_id`) and call `timer_wheel.cancel(id)` on each before the flow-table slot is removed. `cancel()` is idempotent (`tcp_timer_wheel.rs:207-216`), so overlap between the named-handle `Option<TimerId>`s and the `timer_ids: Vec<TimerId>` ledger is a no-op. The `transition_conn` path also handles graceful FIN-driven close without leaking timers because subsequent state transitions feed into the same `outcome.closed` removal.

- **A6.5 hot-path-alloc-elimination did not weaken A5 retransmit correctness.** The two scratch-buffer migrations (`rack_lost_idxs_scratch` at `engine.rs:2980-2988`; `prune_below_into_mbufs` at `tcp_retrans.rs:124-141`) preserve the original semantics: the lost-index list still drives the §6.3 retransmit fan-out before backoff/re-arm; the pruned-mbufs scratch still hands ownership back so the engine can call `shim_rte_pktmbuf_free`. The `SmallVec<[u16; 16]>` indexes at `engine.rs:3018-3023` is a one-shot read-copy out of the `RefCell<...>` scratch so `retransmit()` (which re-borrows the engine) doesn't conflict — correct.

## Test-pyramid concerns

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/rtt_histogram.rs:79-83` — the unit test self-acknowledges a spec/code disagreement at `30000 µs` and treats code as truth.** Spec §7.1 (`docs/superpowers/specs/2026-04-19-stage1-phase-a5-6-rtt-histogram-design.md:193`) asserts `30000 → bucket 11` but the code's `<=` ladder over the §3.2 default edges (`{ ..., 25000, 50000, ... }`) gives `30000 → bucket 12` because `30000 > edges[11]=25000` AND `30000 <= edges[12]=50000`. The test annotation says "treating the code's algorithm result against the §3.2 edges as the source of truth" — but that means the spec's documented expected output remains internally inconsistent and was never reconciled. Either fix spec §7.1 to expect `12`, or change the ladder rule to strict-`<` and update the rest of the test cases. The runtime behavior is consistent with itself; the spec is the broken artifact.

- **No test asserts the `tcp_per_packet_events=false` no-op invariant on the RACK / TLP / RTO emit blocks.** A regression that, say, dropped the runtime gate from the RACK block at `engine.rs:4293` would not be caught by `obs_smoke.rs` or any retrans-tap test that I could find — those tests run with the default config (gate off). A `tcp_per_packet_events_off_emits_zero_per_packet_events_under_loss` integration scenario would close the gap; it's a one-counter assertion (`assert_eq!(events_drained_filtered_by_kind(TcpRetrans), 0)`).

## Observability gaps

- **`obs.events_dropped` overflow soft-cap path is unit-tested in `tcp_events.rs:209-235` (the FIFO + len assertions) but the drop-oldest behavior + counter-bump under sustained pressure is not.** The path at `tcp_events.rs:168-171` (q.len() ≥ soft_cap → pop_front + counter increment) has no direct test of the form "push N+1 events, assert q.len() == N AND `obs.events_dropped == 1`". Closing this would protect against a refactor that reorders the pop_front and the counter inc (e.g. counter inc before pop, double-count on a panic-mid-pop).

- **The `obs.events_queue_high_water` `fetch_max` at `tcp_events.rs:175-178` uses `Ordering::Relaxed`** — correct for the single-lcore RTC model where producer and consumer are the same thread, but a Stage-2 multi-engine merge that aggregates across lcores would silently lose monotonicity. Not a defect today; flag as ARM/Stage-2 follow-up.

## Memory-ordering / ARM-portability concerns

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/rtt_histogram.rs:9 `#[repr(C, align(64))]` hardcodes 64-byte cache-line alignment.** ARM Neoverse-N1 (c7g) is 64 B; ARM ThunderX2 / Apple-silicon performance cores are 128 B. On a 128 B cache-line ARM target, `RttHistogram` would still be 64 B (one half-line) but two adjacent `RttHistogram` instances on the same `TcpConn` cacheline could share a line — false-sharing across the per-conn boundary. Today there's only one per-conn histogram, so the lack of a `cache_line_size!()` portable abstraction does not bite. But the same `#[repr(C, align(64))]` literal appears across the counters module (`counters.rs:6, 113, 131, 307, 779`) — if any of those grows a sibling that shares a struct, the ARM-128 false-sharing exposure surfaces in aggregate. ARM is on the auto-memory roadmap; flag as a single-source-of-truth refactor target before any aarch64 port.

- **`tcp_rtt.rs:47` — `(self.rttvar_us - (self.rttvar_us >> 2)).wrapping_add(delta >> 2)`.** The first operand of `wrapping_add` uses plain subtraction; `x - (x>>2) >= 0` always for `u32` so this cannot underflow. Correct as written.

- **`tcp_input.rs:912, 918, 945` — RTT sampling truncates `now_ns / 1_000` to `u32` µs and uses `wrapping_sub` against the peer's TSecr or our `first_tx_ts_ns / 1_000`.** The TSecr path is symmetric (we sent our `now_us` as TSval; peer echoes it back) so wrap is consistent and the `(1..60_000_000)` sanity bound rejects wrap-confused samples. The Karn's-fallback path at `:945` has the same truncation idiom. The bug_008 retro fix migrated only the RACK age helper (`tcp_rack.rs:159-169`) to u64-ns; the RTT sampler retains the u32-µs scheme but is protected by the 60s sanity gate (a wrap-confused sample lands outside `(1, 60_000_000)` and is rejected). On a long-lived flow at the ~71-min u32-µs boundary, the sampler will silently discard legitimate Karn-RTT samples for a ~60s window once per ~71 minutes — RTT estimator goes stale (RTO defaults stand) but does not produce wrong values. **Not a blocker; sibling-defect to bug_008 not covered by the existing fix.** Worth a follow-up to migrate the RTT sampler to u64-ns for symmetry; the spec §3.2 + RFC 6298 §3 implementations would not change.

## Hidden coupling

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_rack.rs:6 `use crate::tcp_retrans::RetransEntry;`** — RACK reaches into the retransmit module to pattern-match on entry fields. This is fine in Stage 1 (single crate, RTC model), but it inverts the natural layering ("RACK detects loss; the retransmit queue is one consumer of that signal"). The §6.3 helpers `rack_mark_losses_on_rto` / `_into` take `&VecDeque<RetransEntry>` directly rather than via a trait; a Stage-2 multi-CC-mode refactor that grew an alternate retrans-queue shape would have to update RACK first. Acceptable today; flag as Stage-2 abstraction debt.

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_conn.rs:551-561 `tlp_config()` translates the per-conn `u32::MAX` "no-floor" sentinel back to `0` for `TlpConfig.floor_us`.** This translation lives in `TcpConn`, not in the `dpdk-net` ABI translator (which is the canonical sentinel-substitution site at `crates/dpdk-net/src/lib.rs:919-921` for the `0 → tcp_min_rto_us` substitution but does NOT translate the `u32::MAX` sentinel — it stores it verbatim and lets `tlp_config()` translate downstream). Two-stage substitution is split across crate boundaries; either centralize at the ABI layer (so the core sees only post-substitution values, never `u32::MAX`) or document that the core must always translate. Today the split works because `tlp_config()` is the single use site, but a Stage-2 site that also reads `tlp_pto_min_floor_us` directly would silently get `u32::MAX` and project it as a 71-min floor.

## Documentation drift

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/engine.rs:6556-6558` — `Drop for Engine` Step 1 comment says clearing the flow_table "drops every TcpConn, which drops recv-segment + reorder + snd_retrans + delivered_segments mbufs back into still-alive mempools."** Accurate for `delivered_segments` (uses `MbufHandle` which has a refcount-decrementing `Drop`). NOT accurate for `snd_retrans` — `RetransEntry.mbuf` is `crate::mempool::Mbuf`, a non-owning ptr-copy with no `Drop`. On engine teardown, snd_retrans mbufs do not get their refcounts decremented; they're reclaimed only because the mempool itself is about to be torn down. Functionally fine (no UAF, no leak across engine lifetimes — the mempool tear-down reclaims everything), but the comment misrepresents the mechanism. Either rename `Mbuf` to `MbufRef` to make the non-owning shape explicit, or update the comment to acknowledge `snd_retrans` mbufs are pool-reclaimed not ref-decremented.

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_timer_wheel.rs:1-8` module doc says "A5 internal; A6 adds public timer API on top."** A6 has now landed (`engine.rs:2887` `ApiPublic` fire path; `engine.rs:5670` style `dpdk_net_timer_add` plumbing), so "A6 adds" is past-tense drift. Minor.

- **`/home/ubuntu/resd.dpdk_tcp/.claude/worktrees/cross-phase-retro-review/crates/dpdk-net-core/src/tcp_tlp.rs:50-51 `Default::default() = a5_compat(5_000)` hard-codes the A5 default `min_rto_us` floor.** A5.5 made `floor_us` engine-configurable via `EngineConfig.tcp_min_rto_us`. Any code path that constructs a default `TlpConfig` (no current production call site, but the test corpus uses `Default::default()`) gets the 5ms hard-coded value rather than the engine-configured one. Documentation is honest ("`a5_compat` and `default()` preserve A5 behavior exactly so pre-A5.5 tests / call-sites keep working") but the `Default` impl is now exclusively for tests. Either gate `Default` behind `#[cfg(test)]` or move it to a `#[cfg(test)] mod test_helpers` module so production code cannot accidentally pick up the A5-hardcoded floor.

## FYI / informational

- **`tcp_events.rs:13-25 LossCause` enum is `Rack=0, Tlp=1, Rto=2`** with the encoding documented in the variant doc-comment; the C-ABI `dpdk_net_event_tcp_loss_t.trigger` is a `u8`. Repr is `derive(Debug, Clone, Copy, PartialEq, Eq)` with no `#[repr(u8)]`. Today `as u8` casts in the ABI translator produce 0/1/2 because Rust's discriminant-assignment for fieldless enums is sequential, but a future refactor that inserts `Tcp::Bbr` between `Rack` and `Tlp` would silently shift the encoding. **Add `#[repr(u8)]` to pin the wire encoding.**

- **`rtt_histogram.rs:23-35 select_bucket` uses `<=` (inclusive lower bucket on edge match).** This is the rule the test corpus and the per-conn-histogram-coverage sibling audit (`crates/dpdk-net-core/tests/per-conn-histogram-coverage.rs`) consistently encode. Spec §3.1 source-code block shows the same `<=` rule but spec §7.1 expected output for `30000 µs` is inconsistent with the §3.2 default edges + §3.1 ladder rule. See Test-pyramid concerns above.

- **No `cache_line_size!()` macro or `RTE_CACHE_LINE_SIZE` constant is consumed anywhere in core.** Every alignment is a hard `align(64)` literal (counters.rs has 5 sites; rtt_histogram.rs has 1). On x86_64 this is correct; ARM Neoverse-N1 is also 64 B; ARM ThunderX2 / Apple silicon is 128 B. ARM portability roadmap calls this out — flag here for completeness because the RTT histogram is the newest 64-byte-aligned struct (A6 task 6, 18) and adopting a portable abstraction now is cheaper than retrofitting later.

- **`tcp_timer_wheel.rs:259-269 `level_and_bucket_offset` clamps `off` at `BUCKETS - 1 = 255`.** Combined with `cursors[level] + off % BUCKETS`, a delay landing exactly on a level boundary (e.g. delay_ticks = 256 lands at level 1, off=1) is fine, but the `clamp(1, BUCKETS-1)` lower bound of `1` ensures no timer ever lands in the cursor's CURRENT bucket — which would otherwise fire on the same `advance` call that armed it. Correct semantics; called out only because the rationale isn't in the file.
