**Reviewer:** general-purpose subagent (opus 4.7) — covering for superpowers:code-reviewer
**Reviewed at:** 2026-05-05
**Part:** 6 — Test infrastructure (loopback + packetdrill shim + tcpreq + obs gate + coverage expansion)

## Verdict

The test-infrastructure stack is architecturally clean — a coherent `test-server` cargo feature gate funnels both the shim runner and tcpreq-runner through the same `test_packet` builders, and the A8 observability smoke test pins exact-counter values that will trip on any drift. However, the gateway-ARP regression class is only HALF fixed: `test-server` workspace-unification is gated everywhere it needs to be, but `scapy-fuzz-runner` still pulls `dpdk-net-core/test-inject` as a non-optional, non-default-gated workspace dependency, leaving `test-inject` workspace-unified — the exact same architectural pattern that caused both T11 and T20 regressions. A handful of smaller drifts: a `dpdk_net_shutdown` symbol declared in BOTH `dpdk_net.h` and `dpdk_net_test.h`, a redundant inner `#[cfg(feature = "test-server")]` inside an already-gated impl block, an unused `wall_timeout` parameter in the shim invoker (T11+ promise stale), and the test-server-only `inject_rx_frame` quietly diverging in counter-bump semantics from the test-inject variant (now patched, but the dual-implementation pattern remains a future-foot-gun).

## Architectural drift

- **AD-1 — `test-inject` workspace-unification half-fix.** The gateway-ARP regression class (commits `f6280ab` for tcpreq-runner, `8147404`/`50a2392` for layer-h-correctness) was specifically about cargo's workspace feature unification activating `test-server` on `dpdk-net-core` for ALL workspace consumers. The fix was: make every test-only consumer's `dpdk-net-core` dep optional + gate `test-server` behind a non-default feature. `tools/scapy-fuzz-runner/Cargo.toml:8` still violates this:
  ```toml
  dpdk-net-core = { path = "../../crates/dpdk-net-core", features = ["test-inject"] }
  ```
  No `optional = true`, no feature gate. Verified at HEAD with `cargo metadata --no-deps`: scapy-fuzz-runner unconditionally requests `test-inject`, which workspace-unifies into every other `dpdk-net-core` consumer in `cargo build --workspace`. `test-inject` does NOT reroute `tx_frame` (so the immediate ARP-resolve symptom does not reproduce), but it DOES enable a separate `inject_rx_frame` implementation at `crates/dpdk-net-core/src/engine.rs:6302` whose counter-bump semantics quietly differ from the `not(feature = "test-inject")` variant at `engine.rs:6692`. The A8.5 T10 follow-up at `engine.rs:6361` is exactly the bug `crates/dpdk-net-core/tests/counter-coverage.rs:88-99` documents: `eth.rx_pkts` was missing from the test-inject variant, breaking `cover_eth_rx_pkts` under workspace builds. Recommend: same fix shape applied to scapy-fuzz-runner (`optional = true` + non-default `test-inject` feature gate), so production-path workspace builds carry zero `test-inject` code.

- **AD-2 — Two `inject_rx_frame` implementations diverging by feature-gate.** `crates/dpdk-net-core/src/engine.rs:6302` (`#[cfg(feature = "test-inject")]`) and `engine.rs:6692` (`#[cfg(not(feature = "test-inject"))]` inside the `#[cfg(feature = "test-server")]` impl block at `engine.rs:6627`) are two distinct functions with subtly different mempool-source + error-typing + counter-bump behavior. Comments at `engine.rs:6685-6690` describe this as "step aside; superset" but they are NOT a strict superset: the test-inject variant uses a separate `test_inject_pool` (lazy), the test-server-only variant uses `_rx_mempool` (the engine's regular RX pool). Counter-bump parity was lost-then-restored (A8.5 T10 followup at `engine.rs:6360`) — that the fix had to be back-applied is itself the smell. Better shape: a single `inject_rx_frame` body parameterized over which mempool to source from, with both feature flags feeding the same code path.

- **AD-3 — `test-server` impl block + redundant inner `#[cfg]`.** `crates/dpdk-net-core/src/engine.rs:6627` opens an entire `impl Engine` block under `#[cfg(feature = "test-server")]`. Inside, line 6673 has another `#[cfg(feature = "test-server")]` on `pub fn conn_peer_mss` — already implied by the outer block. Stale duplication. Cosmetic, but it suggests the author wasn't sure the outer gate covered them.

- **AD-4 — `dpdk_net_shutdown` declared in BOTH headers.** A8.5 T7 added `dpdk_net_shutdown` as a production-side FFI (full-close only); cbindgen emits it into `include/dpdk_net.h:772` correctly. The same prototype ALSO appears in `include/dpdk_net_test.h:108` because the test-header cbindgen pass (under `--features test-server`) does not exclude production symbols — only `dpdk_net_test_*` symbols are in the exclude list (`crates/dpdk-net/cbindgen.toml:35-51`). Result: a C++ test binary that includes both headers gets a duplicate declaration. Not an ABI break (declarations are identical), but pure header noise. Either suppress production symbols from the test-only header pass, or document that consumers must include exactly one.

- **AD-5 — Test-server bypass behavior is a third gate the spec doesn't describe.** Same drift as Part 5 AD-2 but A7 is the introducer: `crates/dpdk-net-core/src/engine.rs:1299-1311` makes `port_id == u16::MAX` a sentinel that bypasses every `rte_eth_*` call AND zeros every offload-active latch. The test-server spec (`docs/superpowers/specs/2026-04-21-stage1-phase-a7-loopback-test-server-packetdrill-shim-design.md`) introduces `port_id = u16::MAX` but does not name it as the architectural bypass — readers must infer it from comments at the code site. Document the sentinel as a first-class concept (e.g. `port_id_is_synthetic()` helper) so dependent code knows what behavior changes.

## Cross-phase invariant violations

- **CPI-1 — Gateway-ARP regression: missing test that would have caught both incidents.** Commit `f6280ab` (tcpreq-runner) and `8147404` / `50a2392` (layer-h-correctness) tell the same story: a workspace-wide build silently unified `test-server` and rerouted every `tx_frame` to `test_tx_intercept` (engine.rs:2050), gateway ARP requests never reached the wire, and every DPDK-based bench-* binary timed out at 10s with "gateway ARP did not resolve within 10s". Both regressions surfaced in production benchmarks (T11 / T20 bench runs), not in CI. There is NO test in `crates/dpdk-net-core/tests/` or `tools/*/tests/` that asserts `cargo build --workspace --release` produces a `dpdk-net-core` resolve WITHOUT `test-server`. A 5-line CI guard would catch this regression class:
  ```sh
  cargo metadata --no-deps --format-version 1 | jq -e '.packages[] | select(.name=="dpdk-net-core") | .features | any(. == "test-server")' && exit 1 || exit 0
  ```
  AD-1 above confirms the regression class is still latent for `test-inject`. Recommend adding the metadata check to CI for `test-server`, `test-inject`, `test-panic-entry`, `fault-injector`, and `obs-none` — every "should never appear in default-features build" feature.

- **CPI-2 — A8 exact-counter assertions held; A10 perf cherry-picks did NOT drift them.** `crates/dpdk-net-core/tests/obs_smoke.rs:72-121` pins exact values for 21 counters across the connect → 4-send → 1-RTO-retrans → close → 2×MSL scenario; the `assert_no_unexpected_counters` walk at `obs_smoke.rs:288-300` enforces zero on every other declared counter. Inspected at HEAD: each value still has a coherent comment explaining the expected count (e.g. `tcp.tx_flush_bursts = 4` for the 4 send-and-flush cycles), and the `tcp_per_packet_events` default-OFF comment at line 24-28 explains why `TcpRetrans` / `TcpLossDetected` events stay out of the expected event list. No drift detected — the contract holds.

- **CPI-3 — `state_trans[11][11]` exhaustive table at `obs_smoke.rs:152-158` is correct but only covers a single FSM trajectory.** The 121-cell sweep on lines 182-196 enforces zero on unreached cells — strong fail-loud signal. But only 5 of 121 cells are exercised, so the matrix tests "no other transitions fired" (good) but doesn't cover the OTHER 33+ `Reached` transitions that A8 T8 enumerates in `counter-coverage.rs`. The two tests are complementary, not duplicative. The `obs_smoke.rs` framing as "Stage 1 ship-gate counter-drift tripwire" undersells what it does (one trajectory) and oversells what's covered (one trajectory of 33+).

## Tech debt accumulated

- **TD-1 — `wall_timeout` parameter is dead code in the shim invoker.** `tools/packetdrill-shim-runner/src/invoker.rs:30` writes `let _ = wall_timeout;` and pins `timed_out: false` on line 49. The signature comment at line 25 promises real timeout enforcement; the comment at line 30 says "T11+ wires a real timeout via wait_timeout / kill" — but A7 T11 (commit `8d8d97a`) shipped without wiring it, and nothing since has added it. A misbehaved subprocess will hang `cargo test` indefinitely. Mitigation: the CI scripts (`scripts/ci-packetdrill-corpus.sh:19-28`) wrap with `timeout 300/900` outside the cargo invocation, so production CI is safe. Local-dev `cargo test` runs without that wrapper. Either implement `wait_timeout`, or drop the parameter + drop the misleading docstring.

- **TD-2 — `clock.rs:79` TODO on `CLOCK_MONOTONIC_RAW`.** `crates/dpdk-net-core/src/clock.rs:79` (introduced pre-A7 but visible to test-server scope): `// TODO(spec §7.5): spec mandates CLOCK_MONOTONIC_RAW; Rust's Instant::now()`. Test-server scenarios run on virtual time so the TODO doesn't affect them; production path uses `Instant::now()` which is `CLOCK_MONOTONIC` (subject to NTP slewing). Spec deviation, recorded as TODO at the site only.

- **TD-3 — Duplicated `test_eal_args` + `test_server_config` between tcpreq-runner and dpdk-net-core/tests/common.** `tools/tcpreq-runner/src/lib.rs:181-207` openly comments "Duplicated from `tests/common::test_eal_args`" / "Duplicated from `tests/common::test_server_config`". The duplication is a side-effect of cargo not letting an out-of-crate consumer reach into another crate's `tests/common` module. The right shape would be: lift these helpers to a tiny shared crate (`crates/dpdk-net-core/src/test_fixtures.rs` already exists and is gated under `test-inject`). Acceptable for now (drift risk is low — the constants are stable), but each new test-server consumer adds another duplicate.

- **TD-4 — Test-only `dev-dependencies = { dpdk-net-core = { path = ... } }` in `tools/packetdrill-shim-runner/Cargo.toml:28` is non-optional.** Comment at line 22-28 explains why it's needed (the two shim direct self-tests reach for `Engine::inject_rx_frame` / `drain_tx_frames` / `pump_timers`). It IS a dev-dep so it doesn't affect `cargo build --workspace --release` (regression-safe). But for `cargo test --workspace`, this dev-dep activates `dpdk-net-core` without test-server, so the two self-tests at `tools/packetdrill-shim-runner/tests/shim_inject_drain_roundtrip.rs:17` and `shim_virt_time_rto.rs:18` rely on `cfg(feature = "test-server")` cargo feature unification from elsewhere in the workspace to pick up the symbols they import. Brittle: if every other workspace member dropped its `test-server` activation, these tests would fail to compile. Either gate the dev-dep behind a feature, or add `#![cfg(feature = "test-server")]` to the two test files explicitly.

- **TD-5 — A8.5 T9 soak test is `#[ignore]` and only runs with `LIGURIO_SOAK_ITERS` env.** `tools/packetdrill-shim-runner/tests/corpus_ligurio.rs:112` `ligurio_no_crash_soak`. Acceptable (soak tests are by definition opt-in), but no scheduled CI job runs it: `scripts/ci-packetdrill-corpus.sh` only runs the single-iteration version. The crash-safety claim in `tools/packetdrill-shim/SKIPPED.md` ("soak-tested 100×") is therefore an as-of-T9-author claim, not a continuously-verified one.

## Test-pyramid concerns

- **TPC-1 — A8.5's 10 added tasks are largely real coverage, NOT rubber-stamping.** Each ported tcpreq probe has both a probe body in `tools/tcpreq-runner/src/probes/<name>.rs` (200-500 lines including doc-comments + RFC citations) AND an integration test in `tools/tcpreq-runner/tests/probe_<name>.rs` that asserts on `ProbeStatus` (the observable contract), not on internal state. The MUST clauses they cover (MUST-2/3 zero-checksum, MUST-4 OptionSupport, MUST-6 UnknownOption, MUST-7 IllegalLength, MUST-14 MSSSupport, RstAck reset-processing) are real RFC clauses and the probes drive them through the public API surface. Verdict: meaningful, not rubber-stamping.

- **TPC-2 — But the probe tests themselves are 24-47 lines each + assert only on `ProbeStatus::Pass`.** They depend entirely on the probe-body's internal assertions to validate behavior. If a probe implementation gets a false-positive (e.g. counts wrong because it asserted only that `tcp.rx_bad_csum > 0` instead of `== 1` or `+= 1`), the test would still pass. Suggest: at least one negative-shape integration test per probe (e.g. mutate the engine to NOT bump `rx_bad_csum`, assert the probe returns `ProbeStatus::Fail`). Currently zero negative tests.

- **TPC-3 — Counter-coverage uses `bump_counter_one_shot` for HW-only counters.** `crates/dpdk-net-core/tests/counter-coverage.rs:264-450` covers ~15 hardware-path-only counters by calling `bump_counter_one_shot` then asserting the bump. This is the same Part 5 CPI-2 finding: it's a coverage gate (counter is addressable via `lookup_counter`), not a behavior test. The header comment at `counter-coverage.rs:41-52` is honest about the triage; the concern is downstream readers may believe these counters are end-to-end tested when they're only addressability-tested. Real-path coverage lives in `tests/ahw_smoke_ena_hw.rs` (`#[ignore]` by default) and `tests/ena_obs_smoke.rs`.

- **TPC-4 — Corpus tests pin counts (`counts::LIGURIO_RUNNABLE_COUNT` etc.) rather than enumerate scripts.** `tools/packetdrill-shim-runner/tests/corpus_ligurio.rs:89-94` asserts `assert_eq!(runnable.len(), counts::LIGURIO_RUNNABLE_COUNT)`. This catches drift (someone adds/removes a script) but doesn't tell you WHICH script regressed when the count moves. The `runnable` Vec is right there — a `Vec<PathBuf>` log on diff would help triage. Minor ergonomics, not a correctness gap.

## Observability gaps

- **OG-1 — The exact-counter contract in `obs_smoke.rs` does not cover EVERY §9.1 counter.** Out of 95+ counters in `ALL_COUNTER_NAMES`, only 21 have explicit non-zero expected values; the rest are pinned to zero by `assert_no_unexpected_counters`. This is exactly the spec promise (§9.1.1 "every declared `AtomicU64` counter (exact value); non-listed MUST be 0"). What's NOT covered is: A10's perf-instrumentation counters (`obs.events_dropped` is gated zero, `obs.events_queue_high_water = 7` is pinned but the limit-detection counter family from A10 isn't included). If A10 added counters that fire in a 4-send/1-RTO scenario, those should appear in `EXPECTED_COUNTERS` or be confirmed-zero. Worth a one-time audit: walk every counter added since A8 (i.e. roadmap A9, A10, A10.5) and confirm each is either in `EXPECTED_COUNTERS` with an expected value, or expected-zero.

- **OG-2 — `EXPECTED_COUNTERS_OBS_BYTE` is feature-gated at the array literal level (`obs_smoke.rs:131-134`).** Conditional-compilation inside an array literal is fine but fragile: any added entry must remember to gate. Cleaner shape would be a function `fn expected_obs_byte_entries() -> Vec<(&str, u64)>` that returns `vec![]` when the feature is off.

## Hidden coupling

- **HC-1 — `tools/tcpreq-runner` and `dpdk-net-core/tests/common` each have their own `ENGINE_SERIALIZE` Mutex.** `tools/tcpreq-runner/src/lib.rs:173` and `crates/dpdk-net-core/tests/common/mod.rs:441` are two separate `static Mutex<()>`. They do NOT serialize against each other. This is OK in practice because cargo runs each `cargo test -p` invocation in its own process, and DPDK EAL is process-local — two processes share no mempool namespace. But the comment at `tcpreq-runner/lib.rs:146-159` describes the lock as "binary-wide", which is correct, but the implication that "DPDK arena pressure cross-binary" requires care is exactly what `crates/dpdk-net-core/tests/counter-coverage.rs:140-156` documents in its retry-loop fallback. Two separate locks for the same architectural problem with no shared abstraction. Recommend: either lift the lock into a tiny shared `dpdk-test-fixtures` crate, or document at both sites that the lock is per-process and per-binary by design.

- **HC-2 — `tools/packetdrill-shim-runner/tests/*` reach into `dpdk_net_core::test_server::test_packet`, `dpdk_net_core::test_tx_intercept`, `dpdk_net_core::clock`.** All public crate-level surface, all under `#[cfg(feature = "test-server")]`. Not hidden coupling — it's the expected coupling: A7 T12 specifically lifted these helpers to `pub` so out-of-crate consumers (the shim runner) can build wire-identical frames. The contract is honored.

- **HC-3 — `tools/layer-h-correctness` reaches `Engine::state_of` (`engine.rs:6659`).** Public method under `#[cfg(feature = "test-server")]` impl block. Layer-H reads the FSM state per scenario. This is the public test-only contract; not coupling beyond what `state_of`'s docstring promises. Same for `conn_peer_mss` (`engine.rs:6674`) used by the MissingMSS probe.

- **HC-4 — Two public RX-injection paths share a confusing feature predicate.** `Engine::inject_rx_frame` exists with `#[cfg(feature = "test-inject")]` (returns `InjectErr`) and `#[cfg(all(feature = "test-server", not(feature = "test-inject")))]` (returns `crate::Error`). Out-of-crate callers writing tests must know which build of `dpdk-net-core` they're linking against to guess the right error type. The `expect(...)` workaround documented at `engine.rs:6688-6690` ("test-server tests use `.expect(..)` which works with either error type") is fragile. Tighter design: one public `inject_rx_frame` that lives at the same path under both features, with a single error type.
